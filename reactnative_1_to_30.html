Module 1 · Architecture & Setup

This module covers the foundation of every React Native project. You'll learn how the framework communicates with native code, how to choose between Expo workflows, how EAS handles cloud builds and over-the-air updates, and how to read key configuration files. These are the first questions interviewers ask — nail them and you establish instant credibility.

01

What is React Native, and how is it different from a regular React web app?

📖 Answer

React Native is a framework by Meta that lets you build real native mobile apps for iOS and Android using JavaScript and React. In a regular React web app, your components compile down to HTML elements inside a browser. In React Native, those same components map to actual native UI widgets — a <View> becomes a real iOS UIView or Android android.view.View. There is no browser, no WebView, no HTML. You get genuine native rendering and performance from a single JavaScript codebase.

JSX · App.jsx 
import { View, Text, StyleSheet } from 'react-native';

// <View>  →  UIView (iOS) / android.view.View (Android)  — NOT a <div>!
// <Text>  →  UILabel (iOS) / TextView (Android)           — NOT a <p>!
export default function App() {
  return (
    <View style={styles.container}>
      <Text style={styles.heading}>Hello, React Native!</Text>
    </View>
  );
}
const styles = StyleSheet.create({
  container: { flex: 1, justifyContent: 'center', alignItems: 'center' },
  heading:   { fontSize: 22, fontWeight: '700' },
});
02

What is the difference between React Native CLI and Expo?

📖 Answer

React Native CLI is the official bare-bones tool from Meta. It gives you full access to native folders (android/ and ios/) from day one, but requires Android Studio and Xcode installed locally to build anything. Think of it like building your own car from parts — total control, complex setup. Expo is a higher-level platform built on top of RN CLI. It provides a managed environment, pre-built libraries, cloud builds (EAS), and an instant preview app (Expo Go). Think of it as a ready-to-drive rental car. For beginners, Expo is almost always the right starting point.

Bash · Terminal 
# ── Expo (recommended for beginners) ──────────────────────
npx create-expo-app@latest MyApp
cd MyApp
npx expo start         # scan QR in Expo Go → runs instantly! 🎉

# ── React Native CLI (full native control) ─────────────────
npx @react-native-community/cli@latest init MyApp
cd MyApp
npx react-native run-android   # needs Android Studio + emulator
03

What is the Expo Managed Workflow?

📖 Answer

In the Managed Workflow, Expo hides the android/ and ios/ native folders entirely. You only write JavaScript; Expo handles all native code, configuration, and build settings behind the scenes. It's like renting a fully-managed apartment — you just move in and live. All app configuration (icons, name, permissions, splash screen) lives in a single app.json file. Perfect for most apps and all beginners — you get fast iteration without ever touching native code.

JSON · app.json (Managed) 
{
  "expo": {
    "name":    "MyApp",
    "slug":    "my-app",
    "version": "1.0.0",
    "icon":    "./assets/icon.png",
    "splash":  { "image": "./assets/splash.png", "resizeMode": "contain" },
    "ios":     { "bundleIdentifier": "com.yourname.myapp" },
    "android": { "package": "com.yourname.myapp" }
  }
}
// No android/ or ios/ folders — Expo manages them in the cloud ✅
04

What is the Expo Bare Workflow?

📖 Answer

The Bare Workflow is triggered by running expo prebuild. Expo generates the actual android/ and ios/ native project folders from your app.json configuration. You now own and can directly edit those folders — add native libraries, change Gradle settings, edit Info.plist, or write your own native modules. You still benefit from Expo's SDK and EAS services. Think of it as buying your apartment instead of renting — more control and flexibility, but you're now responsible for maintenance.

Bash · Terminal 
# Option A: Start a new bare-workflow project from scratch
npx create-expo-app --template bare-minimum MyBareApp

# Option B: Migrate an existing Managed project to Bare
cd MyApp
npx expo prebuild         # generates android/ and ios/ from app.json
npx expo prebuild --clean # re-generates (overwrites) native folders

# You now have full android/ and ios/ folders to edit directly 🔧
05

As a beginner, when should you choose Managed Workflow, Bare Workflow, or React Native CLI?

📖 Answer

Choose Managed Workflow for 90% of projects — it handles native complexity for you, supports all common device features through Expo's SDK, and gets you shipping fast. Choose Bare Workflow only when you need a native library that isn't available in Expo's SDK (e.g., a custom Bluetooth SDK), but still want to use EAS for cloud builds. Choose React Native CLI when you're joining an existing CLI codebase, or when your team needs the lowest-level native control. The guiding rule: start Managed, upgrade to Bare only when you hit a wall.

Decision Guide 
Managed Workflow  🟢  Best for: beginners, most production apps, fast builds
                       You get: Expo Go preview, EAS cloud builds, no native code
                       
Bare Workflow     🟡  Best for: custom native libs + still want EAS services
                       You get: android/ ios/ folders + full Expo SDK
                       
React Native CLI  🔴  Best for: legacy codebase, enterprise, zero Expo
                       You get: maximum control, maximum setup complexity
06

What is the Metro Bundler and what role does it play?

📖 Answer

Metro is React Native's JavaScript bundler — the tool that collects all your JS files, resolves imports, applies Babel transforms, and packages everything into a single optimized bundle your app can execute. Think of Metro as a logistics truck: it drives to every import in your project, picks up each file, and delivers one consolidated package to your device. During development Metro runs a local web server so your phone fetches the latest bundle over Wi-Fi on every change. In production the bundle is compiled and embedded inside the app binary.

Bash · Terminal 
# Metro starts automatically with any of these commands:
npx expo start                   # → Metro on exp://192.168.x.x:8081
npx react-native start           # → Metro on http://localhost:8081

# Metro's responsibilities:
# 1. Resolve module imports (node_modules, aliases)
# 2. Transform JSX + modern JS via Babel
# 3. Bundle files into a single JS bundle
# 4. Serve bundles over HTTP to the device / emulator
# 5. Enable Hot Module Replacement (Fast Refresh)
07

What was the "Bridge" in the old React Native architecture, and why was it a bottleneck?

📖 Answer

In the original architecture, JavaScript and native code ran in completely separate threads and couldn't call each other directly. They communicated through a Bridge — an asynchronous messaging layer that serialized all data to JSON before passing it across, then deserialized it on the other side. Think of two offices passing notes through a slow mail room. Because every message (layout calculation, gesture event, animation frame) had to be JSON-encoded and decoded, high-frequency interactions like smooth animations could overwhelm the bridge and cause dropped frames. This is why the New Architecture was introduced.

Conceptual Diagram 
── OLD ARCHITECTURE (Bridge) ─────────────────────────────────
  JS Thread  ──[serialize to JSON]──►  Bridge  ──[deserialize]──►  Native Thread
             ◄──[serialize to JSON]──          ◄──[deserialize]──
  Problem: asynchronous, high-volume messages cause congestion

── NEW ARCHITECTURE (JSI + Fabric) ──────────────────────────
  JS Thread  ◄──────────  JSI (C++ direct function refs)  ──────────►  Native
  Benefit: synchronous calls possible, no JSON overhead, zero copy
08

What is the New Architecture in React Native — JSI, Fabric, and Turbo Modules?

📖 Answer

The New Architecture replaces the Bridge with three pillars. JSI (JavaScript Interface) is a lightweight C++ layer that lets JavaScript hold direct references to native objects and call native functions without JSON serialization — like giving JS and native code a shared phone line. Fabric is the new concurrent rendering engine for UI components. Turbo Modules are the new native module format — they load lazily (only when first accessed) instead of all at startup, dramatically improving launch time. This is enabled by default in React Native 0.76+ and Expo SDK 52+.

Config · Enabling New Arch 
// In Expo app.json (enabled by default in SDK 52+):
{
  "expo": {
    "newArchEnabled": true
  }
}

// In React Native CLI — android/gradle.properties:
newArchEnabled=true

// In React Native CLI — ios/Podfile:
# New Architecture is ON by default in RN 0.76+
# No extra config needed in new projects.

What is the Hermes JavaScript engine and why does React Native use it?

📖 Answer

Hermes is a JavaScript engine built by Meta specifically for React Native. Traditional engines like V8 or JavaScriptCore compile your JS code Just-in-Time at runtime — like a chef cooking from scratch every service. Hermes takes a different approach: it pre-compiles JavaScript to bytecode at build time, so at startup the engine just executes the pre-prepared bytecode — like reheating a pre-cooked meal. The result is significantly faster app launch times, lower peak memory usage, and better battery performance. Hermes is the default engine in Expo SDK 48+ and React Native 0.70+.

JSON · app.json 
// Hermes is the default in Expo SDK 48+ and RN 0.70+
// You can explicitly set it in app.json:
{
  "expo": {
    "jsEngine": "hermes"    // ✅ recommended (default)
    // "jsEngine": "jsc"    // JavaScriptCore (legacy fallback)
  }
}

// Hermes benefits vs. JSC:
// ✅ ~50% faster Time-To-Interactive on cold start
// ✅ Lower RAM usage (~30-40% in benchmarks)
// ✅ Smaller binary size for JS payload

How do you create and run a brand-new Expo project from scratch?

📖 Answer

You need Node.js v18+ installed, then use create-expo-app which scaffolds a complete project with all dependencies, a working App.tsx, and Expo configuration. Running npx expo start launches the Metro bundler and displays a QR code. Open the free Expo Go app on your physical Android or iOS device, scan the QR code, and your app appears on screen within seconds — no emulator, no native build required.

# Step 2 — enter the directory

# Step 3 — start the dev server + Metro bundler

# Step 4 — on your phone:

#   iOS:     Open Camera  → Tap the Expo Go link
# Your app is now live on your device! 📱

What is the difference between app.json and app.config.js in Expo?

📖 Answer

app.json is a static JSON file with fixed, hardcoded values — simple and works perfectly for most single-environment apps. app.config.js is the dynamic alternative — a JavaScript file that exports a config object, so it can read process.env variables, execute logic, and return different settings for different environments (development, staging, production). As a beginner, app.json is perfectly fine. Graduate to app.config.js when you need multi-environment configuration.

JS · app.config.js 
// app.config.js — dynamic; can read env variables at build time
export default ({ config }) => ({
  ...config,                                  // spread existing app.json values
  name:    process.env.APP_ENV === 'prod' ? 'MyApp' : 'MyApp [DEV]',
  slug:    'my-app',
  version: '2.1.0',
  extra: {
    apiUrl: process.env.API_BASE_URL ?? 'http://localhost:3000',
  },
});

What is EAS — Expo Application Services?

📖 Answer

EAS is Expo's cloud platform covering the full mobile app lifecycle. It consists of three main services: EAS Build (compile your app binary in the cloud), EAS Submit (publish to Google Play or Apple App Store), and EAS Update (push JavaScript-only updates to users without a store review). Think of EAS as your entire DevOps team hosted in the cloud — it eliminates the need for a Mac to build iOS apps, manages signing certificates automatically, and lets Windows and Linux developers ship to both platforms equally.

Bash · EAS Setup 
# Install the EAS CLI
npm install -g eas-cli

# Login to your Expo account
eas login

# Initialize EAS in your project (creates eas.json)
eas build:configure

# The three EAS services:
# eas build    → compile native binary (.apk / .ipa) in cloud ☁️
# eas submit   → upload to Google Play Store / Apple App Store 🏪
# eas update   → push JS-only OTA updates (no store review!) ⚡
13

What is EAS Build and why is it valuable?

📖 Answer

EAS Build compiles your React Native project in the cloud into a production-ready native binary — an .apk or .aab for Android, and an .ipa for iOS. The standout benefit: you don't need a Mac to build iOS apps. EAS's cloud servers have Xcode installed; you just push your code and receive a download link when the build finishes. EAS also handles signing certificates and provisioning profiles automatically. Think of sending your recipe to a professional kitchen — you provide ingredients, they produce the finished dish.

Bash · EAS Build 
# Build Android APK for internal QA testing
eas build --platform android --profile preview

# Build iOS IPA for the App Store (needs Apple Dev account)
eas build --platform ios --profile production

# Build BOTH platforms simultaneously
eas build --platform all --profile production

# Check your queued / completed builds:
eas build:list

What is EAS Submit?

📖 Answer

EAS Submit automates uploading your compiled app binary to the Google Play Store or Apple App Store Connect. Normally, submitting requires logging into web portals, filling in metadata, uploading your binary, and navigating complex dashboards. EAS Submit does all of this with a single terminal command. You can even chain it directly onto eas build so your app is built and submitted to the store in one unattended workflow — like having a shipping assistant who delivers straight to the store shelf.

Bash · EAS Submit 
# Submit the most recent Android build to Google Play
eas submit --platform android

# Submit the most recent iOS build to App Store Connect
eas submit --platform ios

# Build AND auto-submit in one single command ✅
eas build --platform android --profile production --auto-submit

# Submit a specific build by ID:
eas submit --platform ios --id <build-id>

What is EAS Update and what are Over-The-Air (OTA) updates?

📖 Answer

EAS Update lets you push changes to your app's JavaScript code and assets directly to users' phones, completely bypassing the app store review process (which can take days). When a user opens your app, it silently checks for a new JS bundle in the background and downloads it. This works because only your JavaScript layer changes — the native binary stays the same. OTA updates are ideal for fixing bugs or updating content instantly. Important limitation: you cannot change native code, SDK versions, or app permissions via OTA — those always require a full store release.

Bash + JSX 
# Publish an OTA update to the production channel
eas update --branch production --message "Fix: cart total calculation"

// Manually check for and apply updates in your app:
import * as Updates from 'expo-updates';

async function checkForUpdate() {
  const { isAvailable } = await Updates.checkForUpdateAsync();
  if (isAvailable) {
    await Updates.fetchUpdateAsync();
    await Updates.reloadAsync();  // restart app with new bundle
  }
}

What is a Development Build in Expo and when do you need one?

📖 Answer

A Development Build is a custom version of Expo Go compiled specifically for your project. The standard Expo Go app contains a fixed set of pre-bundled native libraries. If your app uses a native package not in that set — like react-native-ble-plx for Bluetooth or a custom camera SDK — Expo Go will crash or error. A development build includes your custom native packages bundled together with Expo's developer tools. You install it once as an APK/IPA on your device, then iterate on JS changes instantly via Fast Refresh — the best of both worlds.

Bash · Dev Build Workflow 
# Step 1: Build a development APK via EAS (includes your custom native libs)
eas build --profile development --platform android

# Step 2: Install the APK on your device or emulator
#  (EAS gives you a download link and QR to install)

# Step 3: Start the development server with --dev-client flag
npx expo start --dev-client

# Scan QR → your custom dev build opens (not the public Expo Go app)
# JS changes still hot-reload instantly! ⚡

What is the Expo Go app?

📖 Answer

Expo Go is a free app available on the iOS App Store and Google Play that acts as a universal viewer for Expo projects. You simply run npx expo start, scan the QR code with Expo Go on your phone, and your app launches on a real device within seconds — no compiling, no USB cable, no APK installation. It's the fastest possible way for beginners to see code running on real hardware. The trade-off: Expo Go only supports Expo's built-in library set. It cannot run custom native modules.

Bash · Quick Start Flow 
# 1. Start your project
npx expo start

# 2. A QR code appears in the terminal (and at http://localhost:8081)

# 3. Scan it with your phone:
#    Android → Open the Expo Go app → tap "Scan QR code"
#    iOS     → Open the native Camera app → tap the Expo Go link

# ✅ Your app is now running live on your real device!
# No build step, no emulator, no USB required.

What are the key limitations of Expo Go?

📖 Answer

Expo Go has a fixed, pre-bundled set of native libraries. If your project installs any native npm package not included in that set, Expo Go will throw an error or silently fail. Additionally: (1) you cannot use Expo Go for production or TestFlight distribution; (2) some permissions behave differently inside Expo Go versus a real standalone build; (3) you cannot test custom native modules. For any of these cases, the correct solution is to create a Development Build, which bundles your project's specific native dependencies.

Reference · What Works Where 
✅  Works in Expo Go:
    expo-camera, expo-location, expo-sensors, expo-av
    expo-notifications (with limited features)

❌  Does NOT work in Expo Go:
    react-native-ble-plx        (custom Bluetooth native module)
    react-native-nfc-manager    (NFC hardware access)
    Any custom Java/Swift native module you write yourself

→  Solution for all of the above:
   eas build --profile development --platform android

What can and cannot be changed via an OTA update?

📖 Answer

An OTA update replaces only the JavaScript bundle and static assets inside an already-installed app — it cannot modify the native binary. The rule of thumb: anything that Babel compiled from your .js/.ts/.tsx files can be updated OTA. Anything that required compiling native code (Java, Kotlin, Swift, Objective-C) requires a full new store release. This is why adding a new native npm package (e.g., a Bluetooth library) always requires a fresh EAS Build, not just an OTA update.

Reference Chart 
OTA Update ✅ CAN change:           OTA Update ❌ CANNOT change:
  JavaScript logic & components       Native modules (Java/Kotlin/Swift)
  Screen layouts and styles           New npm packages with native code
  Images, fonts, JSON data files      Permissions in AndroidManifest.xml
  Bug fixes anywhere in JS layer      App icon / splash (native asset)
  Config values in app.config.js      Expo SDK version upgrade
  API endpoints, feature flags        React Native version upgrade

What are the android/ and ios/ folders in a React Native project?

📖 Answer

In React Native CLI and Expo Bare Workflow, your project contains real native project folders. android/ is a full Gradle-based Android Studio project — you can open it in Android Studio, edit AndroidManifest.xml, add native dependencies in build.gradle, and write Java/Kotlin code. ios/ is a full Xcode project — you open .xcworkspace, edit Info.plist for permissions, and run pod install for iOS native dependencies. In Expo Managed Workflow, these folders don't exist — Expo generates them internally during the cloud build process.

Project Tree · Bare / CLI 
MyApp/
├── android/                             ← Full Android (Gradle) project
│   └── app/
│       ├── src/main/AndroidManifest.xml ← permissions, package name
│       ├── src/main/java/...            ← Java/Kotlin native code
│       └── google-services.json         ← Firebase config
├── ios/                                 ← Full iOS (Xcode) project
│   ├── MyApp.xcworkspace                ← open this in Xcode
│   ├── MyApp/Info.plist                 ← iOS permissions
│   └── Podfile                          ← CocoaPods dependencies
├── app/                                 ← Your React Native JS source
└── package.json

What is the purpose of package.json in a React Native project?

📖 Answer

package.json is your project's identity card and dependency manifest. It tells Node.js the project name, version, which npm packages your app needs at runtime (dependencies), which packages are only needed during development (devDependencies), and which short-hand terminal commands you can run (scripts). When a teammate clones your repo and runs npm install, Node reads package.json and downloads every listed package. Without it, your project cannot be set up on a new machine.

JSON · package.json 
{
  "name": "my-app",
  "version": "1.0.0",
  "scripts": {
    "start":   "expo start",
    "android": "expo run:android",
    "ios":     "expo run:ios",
    "test":    "jest"
  },
  "dependencies": {
    "expo": "~53.0.0",         // runtime — shipped with app
    "react": "18.3.2",
    "react-native": "0.76.5"
  },
  "devDependencies": {
    "typescript": "^5.3.0"   // dev-only — not in production bundle
  }
}

What is babel.config.js and what does it do?

📖 Answer

Babel is a JavaScript compiler that transforms modern syntax — JSX, TypeScript, optional chaining (?.), nullish coalescing (??) — into JavaScript that older engines understand. babel.config.js tells Babel which transformation rules (presets and plugins) to apply. For Expo projects, the standard preset is babel-preset-expo, which handles all React Native-specific transforms. If you use Reanimated for animations, you must add its Babel plugin — and it must be the last entry in the plugins array.

JS · babel.config.js 
module.exports = function (api) {
  api.cache(true);
  return {
    presets: ['babel-preset-expo'],  // handles JSX, TS, RN transforms
    plugins: [
      // Optional: add import path aliases
      ['module-resolver', { alias: { '@': './src' } }],
      // ⚠️ react-native-reanimated MUST be the LAST plugin:
      'react-native-reanimated/plugin',
    ],
  };
};

What is metro.config.js and when do you need to edit it?

📖 Answer

metro.config.js configures the Metro bundler's behavior — which file extensions it recognizes, how it resolves modules, which transformers to use for special file types, and how monorepos are structured. For simple projects you never need to touch it. You would edit it when: (1) you want to import SVG files as React components (needs react-native-svg-transformer), (2) you have a monorepo with shared packages, or (3) you need custom module alias resolution at the bundler level rather than via Babel.

JS · metro.config.js 
const { getDefaultConfig } = require('expo/metro-config');

const config = getDefaultConfig(__dirname);

// Example: Enable SVG import as React components
const { transformer, resolver } = config;
config.transformer = {
  ...transformer,
  babelTransformerPath: require.resolve('react-native-svg-transformer'),
};
config.resolver = {
  ...resolver,
  assetExts:  resolver.assetExts.filter(ext => ext !== 'svg'),
  sourceExts: [...resolver.sourceExts, 'svg'],
};
module.exports = config;

What is the difference between a debug build and a release build?

📖 Answer

A debug build is for development: it connects to the Metro dev server over your local network (so Fast Refresh works), includes the Red Box error overlay and DevTools inspector, and runs un-optimized JS. It is significantly slower than production. A release build is what ships to users: JavaScript is fully minified and bundled into the native binary, Hermes pre-compiles it to bytecode, the Metro connection is severed, and all developer tooling is stripped out. Release builds are noticeably faster and must be used for any performance testing.

Bash · Build Variants 
# ── DEBUG (development) ───────────────────────────────────
npx expo run:android              # debug by default
npx react-native run-android      # debug by default
eas build --profile development   # EAS debug build

# ── RELEASE (production) ──────────────────────────────────
npx expo run:android --variant release
npx react-native run-android --mode release
eas build --profile production    # EAS release build → store-ready binary

What is Fast Refresh in React Native?

📖 Answer

Fast Refresh (introduced in React Native v0.61) is the development feature that instantly updates your app when you save a file, without losing component state. Unlike the old "Live Reload" (which restarted the entire app), Fast Refresh surgically re-renders only the components affected by your change. Change a style color and save — you see it update in under a second, with your text inputs, navigation position, and useState values intact. It works out of the box in both Expo and React Native CLI development mode.

JSX · Demo 
// Change 'royalblue' → 'tomato' below, then Save the file.
// Fast Refresh updates the color instantly on your device!
// The count value is PRESERVED — state is not lost. ✅
export default function Counter() {
  const [count, setCount] = useState(0);
  return (
    <Pressable
      onPress={() => setCount(c => c + 1)}
      style={{ backgroundColor: 'royalblue', padding: 20, borderRadius: 12 }}>
      <Text style={{ color: '#fff' }}>Tapped {count} times</Text>
    </Pressable>
  );
}

What are Native Modules in React Native?

📖 Answer

Native Modules are pieces of code written in Java/Kotlin (for Android) or Swift/Objective-C (for iOS) that expose device platform functionality to the JavaScript layer. When React Native doesn't have a pure-JS solution for something — Bluetooth, NFC, biometric authentication, haptic feedback, background audio — native modules bridge that gap. In practice, you rarely write them from scratch; popular ones are pre-packaged as npm libraries (e.g., expo-haptics, expo-local-authentication). Under the New Architecture, native modules are called Turbo Modules and load lazily for better startup performance.

JSX · Consuming Native Modules 
import * as Haptics from 'expo-haptics';
import * as LocalAuth from 'expo-local-authentication';

// expo-haptics internally calls the iOS/Android haptic API (native)
async function onButtonPress() {
  await Haptics.impactAsync(Haptics.ImpactFeedbackStyle.Heavy);
}

// expo-local-authentication calls Face ID / fingerprint (native biometrics)
async function authenticate() {
  const { success } = await LocalAuth.authenticateAsync({ promptMessage: 'Verify identity' });
  console.log('Authenticated:', success);
}

What is the difference between an .apk and an .aab file for Android?

📖 Answer

An .apk (Android Package) is a self-contained, ready-to-install file you can share directly — send it via email or a link, and users tap to install. Perfect for internal QA testing without the Play Store. An .aab (Android App Bundle) is the modern format required by Google Play since 2021. The Play Store uses the .aab to generate device-specific optimized APKs (tailored to screen density and CPU architecture), resulting in 15–50% smaller download sizes for end users. You cannot install an .aab directly on a device.

JSON · eas.json — build types 
{
  "build": {
    "preview": {
      "distribution": "internal",
      "android": { "buildType": "apk" }          // ← .apk for direct install / QA
    },
    "production": {
      "android": { "buildType": "app-bundle" },  // ← .aab for Google Play Store
      "ios":     { "resourceClass": "m-medium" }
    }
  }
}

What is a Provisioning Profile in iOS development?

📖 Answer

A Provisioning Profile is a signed certificate issued by Apple that grants permission for a specific app (identified by its Bundle Identifier) to run on specific devices, use specific entitlements (push notifications, iCloud, Apple Pay, etc.), and be distributed in a specific way (App Store, Ad Hoc, or Enterprise). Think of it as Apple's signed permission slip — without it, iOS will refuse to run your app. The four types are: Development, Ad Hoc, App Store, and Enterprise. EAS handles provisioning profiles completely automatically, which is one of its biggest beginner-friendly benefits.

Bash · EAS Manages This For You

What is google-services.json and where does it go in your project?

📖 Answer

google-services.json is a configuration file downloaded from the Firebase Console (Project Settings → Your Apps → Android). It contains your Firebase project's API keys, project ID, storage bucket, messaging sender ID, and app IDs that let your Android app connect to Firebase services (Firestore, FCM push notifications, Analytics, Auth, etc.). Think of it as the access card your Android app shows to Firebase to prove "I belong to this project." Never commit it to a public GitHub repository — it contains keys that should remain private.

Config · File Placement 
# ── Bare Workflow / React Native CLI ──────────────────────
# Place the file directly in the Android app folder:
android/app/google-services.json   ✅

# ── Expo Managed Workflow ──────────────────────────────────
# Place it at project root and reference in app.json:
{
  "expo": {
    "android": {
      "googleServicesFile": "./google-services.json"
    }
  }
}

# iOS equivalent: GoogleService-Info.plist
# { "expo": { "ios": { "googleServicesFile": "./GoogleService-Info.plist" } } }

What is eas.json and what are build profiles?

📖 Answer

eas.json is the configuration file for EAS Build and EAS Submit. It lives at the project root and defines named build profiles — each a different recipe for building your app. The three standard profiles are: development (debug build with the dev client for rapid iteration), preview (internal APK/IPA without store submission, for QA teams), and production (release AAB/IPA for the app store). You select a profile at build time with the --profile flag. EAS created eas.json automatically when you run eas build:configure.

JSON · eas.json 
{
  "build": {
    "development": {
      "developmentClient": true,       // use dev-client, not Expo Go
      "distribution":      "internal", // install via QR / link
      "android": { "buildType": "apk" }
    },
    "preview": {
      "distribution": "internal",
      "android": { "buildType": "apk" }  // shareable APK for QA
    },
    "production": {
      "android": { "buildType": "app-bundle" },  // .aab for Play Store
      "ios":     { "resourceClass": "m-medium" } // M2 Mac build runner
    }
  }
}
