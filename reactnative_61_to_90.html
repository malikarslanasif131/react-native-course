Module 3 · Ecosystem & Modern Libraries [2026 Standards]

This module covers the libraries that surround the core React Native framework in a real, modern app. You'll learn how screens connect together (navigation), how data and app state flow around your components (state management), and how teams are styling apps in 2026 beyond plain StyleSheet.create(). Knowing this ecosystem signals to interviewers that you can work in a real production codebase, not just a tutorial app.

61

What is React Navigation and what problem does it solve?

📖 Answer

React Native has no built-in concept of "pages" or browser history like the web does. React Navigation is the most established library that fills this gap — it lets you define a set of screens and manages transitioning between them, maintaining a navigation history stack, and handling native back-gesture behavior on both iOS and Android. Think of it as the GPS and road system for your app: it doesn't build the destinations (your screens), but it knows how to get from one to another and remembers where you've been.

Bash · Install 
npx expo install @react-navigation/native @react-navigation/native-stack
npx expo install react-native-screens react-native-safe-area-context
# Wrap your app root in <NavigationContainer> to get started
62

What is a Stack Navigator and how does it work?

📖 Answer

A Stack Navigator organizes screens like a literal stack of cards — each new screen you navigate to is pushed on top, and going back pops the top card off, revealing the one underneath. It automatically provides the native slide-in transition and a back button/gesture, matching platform conventions (iOS slides from the right, Android can use a fade or slide depending on theme). It's the navigator pattern you'll use for the vast majority of "drill-down" flows, like tapping a list item to open a detail screen.

JSX · createNativeStackNavigator 
import { createNativeStackNavigator } from '@react-navigation/native-stack';
const Stack = createNativeStackNavigator();

<Stack.Navigator>
  <Stack.Screen name="Home" component={HomeScreen} />
  <Stack.Screen name="Details" component={DetailsScreen} />
</Stack.Navigator>
63

What is a Tab Navigator (Bottom Tabs)?

📖 Answer

A Bottom Tab Navigator renders a row of icons/labels fixed at the bottom of the screen — tapping one instantly switches to a different top-level section of the app, like Home, Search, and Profile. Unlike a Stack, tabs don't "push" on top of each other; they're siblings you jump between, and each tab typically keeps its own internal navigation state. Think of it as channel buttons on a TV remote — pressing a different button switches the whole picture instantly, and switching back returns you to whatever was last showing on that channel.

JSX · createBottomTabNavigator 
import { createBottomTabNavigator } from '@react-navigation/bottom-tabs';
const Tab = createBottomTabNavigator();

<Tab.Navigator>
  <Tab.Screen name="Home" component={HomeScreen} />
  <Tab.Screen name="Profile" component={ProfileScreen} />
</Tab.Navigator>
64

What is a Drawer Navigator?

📖 Answer

A Drawer Navigator hides a vertical list of navigation links in a panel that slides in from the side — usually triggered by a hamburger icon or an edge swipe gesture. It's a common pattern for apps with many top-level sections that would crowd a bottom tab bar (e.g., Gmail's sidebar with Inbox, Starred, Sent, Trash). Think of it as a kitchen drawer you pull open to find a tool, then push closed again once you've picked your destination.

JSX · createDrawerNavigator 
import { createDrawerNavigator } from '@react-navigation/drawer';
const Drawer = createDrawerNavigator();

<Drawer.Navigator>
  <Drawer.Screen name="Inbox" component={InboxScreen} />
  <Drawer.Screen name="Sent" component={SentScreen} />
</Drawer.Navigator>
65

How do you pass parameters between screens in React Navigation?

📖 Answer

You pass a second argument to navigation.navigate() containing any data you want to send — an object with whatever keys you like (e.g., an item's id). The destination screen reads it back through the route.params object, which every screen component automatically receives as a prop. It's like handing someone a sealed envelope as you point them toward a room — the envelope (params) travels with them and they open it (read route.params) once they arrive.

JSX · Passing & Reading Params 
// Sender screen
navigation.navigate('Details', { itemId: 42, title: 'Cool Item' });

// Receiver screen
function DetailsScreen({ route }) {
  const { itemId, title } = route.params;
  return <Text>{title} (#{itemId})</Text>;
}
66

What is the difference between navigate() and push()?

📖 Answer

navigate() is "smart" — if the target screen already exists somewhere in the current stack, it jumps back to that existing instance instead of creating a duplicate. push() always adds a brand-new instance of the screen onto the stack, even if one already exists, which is useful for cases like a "related products" screen where you genuinely want a fresh copy each time (and a fresh back button per click). It's the difference between returning to a tab you already have open in a browser (navigate) versus always opening a new tab (push).

JSX · navigate vs push 
// Reuses an existing "Profile" screen in the stack if present
navigation.navigate('Profile', { userId: 1 });

// Always stacks a brand-new "Profile" screen instance
navigation.push('Profile', { userId: 2 });
67

What is Expo Router and how is it different from React Navigation?

📖 Answer

Expo Router is built on top of React Navigation, but replaces manually defining <Stack.Screen> components with file-based routing — similar to how Next.js works on the web. Instead of writing navigator config, you simply create files inside an app/ directory, and each file automatically becomes a route. It's the difference between hand-drawing a map of every room in a building (React Navigation) versus the building itself being laid out so the floor plan IS the map (Expo Router) — less boilerplate, and it also enables deep linking and web URLs for free.

File Tree · Expo Router 
app/
├── _layout.tsx       ← root layout, wraps every screen
├── index.tsx         ← matches route "/"
├── settings.tsx      ← matches route "/settings"
└── profile/[id].tsx  ← matches route "/profile/123"
68

How does file-based routing work in the Expo Router app directory?

📖 Answer

Every file placed inside the app/ directory automatically becomes a navigable route — its file path maps directly to a URL-like path. app/index.tsx is the home screen ("/"), app/settings.tsx becomes "/settings", and nested folders create nested paths (app/profile/edit.tsx → "/profile/edit"). There's no separate config file listing your routes — the file system itself is the navigation structure, similar to how a physical filing cabinet's folder structure tells you exactly where to find any document without needing a separate index.

File Tree · Path Mapping 
app/index.tsx          →  "/"
app/settings.tsx       →  "/settings"
app/profile/edit.tsx   →  "/profile/edit"
app/(tabs)/home.tsx    →  "/home"   (parens = layout group, hidden from URL)
69

What are dynamic routes in Expo Router, like [id].tsx?

📖 Answer

Square brackets in a filename create a dynamic segment — a placeholder that matches any value at that position in the URL. A file named [id].tsx inside a profile/ folder will match /profile/1, /profile/abc, or any other value, capturing whatever was typed as a parameter you can read inside the component. It's like a mail slot labeled "Apartment ___" — any apartment number fits, and once a letter arrives you can read exactly which number was written on it.

TSX · app/profile/[id].tsx 
import { useLocalSearchParams } from 'expo-router';

export default function ProfileScreen() {
  const { id } = useLocalSearchParams();   // reads the matched segment
  return <Text>Viewing profile #{id}</Text>;
}
// Visiting /profile/42 renders this with id === "42"
70

What does a _layout.tsx file do in Expo Router?

📖 Answer

A _layout.tsx file defines shared UI or structure that wraps every route inside its folder — similar to a layout template that appears on every page. The root app/_layout.tsx commonly sets up global providers (theme, navigation stack type) that every screen needs. A _layout.tsx inside a subfolder can define a Tab Navigator or Stack Navigator that applies only to the screens in that folder. Think of it as the picture frame that stays constant while different photos (screens) get swapped in and out inside it.

TSX · app/_layout.tsx 
import { Stack } from 'expo-router';

export default function RootLayout() {
  return (
    <Stack>
      <Stack.Screen name="index" options={{ title: 'Home' }} />
    </Stack>
  );
}
71

What is the Context API and when should you use it?

📖 Answer

React's built-in Context API lets you share a value across a whole tree of components without manually passing it down as a prop through every level. You create a Context, wrap your component tree in its Provider with a value, and any descendant can read that value with useContext — no matter how deeply nested. It's best for relatively stable, app-wide data like the current logged-in user, a theme, or a language setting — think of it as a building's central PA system, where any room can hear the announcement without someone walking it floor by floor.

JSX · Context API 
const ThemeContext = createContext('light');

// Provide it once near the top
<ThemeContext.Provider value="dark">
  <App />
</ThemeContext.Provider>

// Read it anywhere below, no prop drilling
const theme = useContext(ThemeContext);
72

What is "prop drilling" and how does Context solve it?

📖 Answer

Prop drilling is the tedious pattern of passing a piece of data down through several layers of components as props, even though only the deepest, final component actually needs it — every component in between must accept and forward it, cluttering their signatures. Context solves this by letting the deeply nested component pull the value directly, skipping all the intermediate "middlemen." It's like passing a note hand-to-hand through five people who don't need to read it (prop drilling) versus the final recipient just checking a shared bulletin board themselves (Context).

Conceptual · Before & After 
// ❌ Prop drilling — user passed through 3 unrelated components
<Header user={user}><Nav user={user}><Avatar user={user} /></Nav></Header>

// ✅ Context — Avatar reads it directly, Header/Nav don't need to know
const user = useContext(UserContext);
73

What are the downsides of Context API for frequently-changing state?

📖 Answer

Every time a Context's value changes, every component consuming that Context re-renders — even ones that only use a tiny unrelated slice of the value, since Context doesn't support partial subscriptions out of the box. For state that updates often (like a scroll position, a typed search query, or live counters), this can cause unnecessary re-renders across large parts of your app. It's like a PA system announcing every tiny update — useful for rare important news, annoying and noisy if used for constant minute-by-minute chatter. This is exactly the gap libraries like Zustand fill.

Conceptual · Re-render Cost 
<CounterContext.Provider value={count}>
  <ComponentA />  {/* re-renders on every count change */}
  <ComponentB />  {/* re-renders too, even if it ignores count */}
  <ComponentC />  {/* re-renders too — no selective subscription */}
</CounterContext.Provider>

What is Zustand and how does it differ from the Context API?

📖 Answer

Zustand is a small, popular state management library that creates a central "store" living outside the React component tree entirely. Components subscribe to only the specific slice of state they actually use via a selector function, so they only re-render when that exact slice changes — solving Context's "everyone re-renders" problem. There's also no <Provider> wrapper needed. Think of it as a shared whiteboard anyone can read from or write to directly, where each person only watches the one line that matters to them, rather than an announcement everyone must hear in full.

Bash · Install 
npm install zustand
# No Provider wrapper required — just import the store hook anywhere
75

How do you create and use a Zustand store?

📖 Answer

You call create() with a function that returns your initial state plus any update functions ("actions") that call set() to modify state. The result is a hook you import and call directly in any component — passing a selector function so the component only subscribes to the piece of state it needs. There's no boilerplate of reducers, action types, or dispatch — just plain functions reading and writing a shared object, which is why beginners often find it far simpler than Redux.

JS · Zustand Store 
import { create } from 'zustand';

const useCounterStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 })),
}));

// In any component — only re-renders when `count` changes
const count = useCounterStore((state) => state.count);
const increment = useCounterStore((state) => state.increment);
76

What is Redux, and why has it become less common for beginners in 2026?

📖 Answer

Redux is an older, highly structured state management library built around a single global store, pure-function "reducers," and dispatched "actions" describing what happened. It's powerful for very large apps with complex state interactions and excellent debugging tools, but the ceremony of action types, action creators, and reducer boilerplate is a lot to learn upfront. For most apps in 2026, lighter alternatives like Zustand (for client state) combined with TanStack Query (for server/async data) cover the same needs with far less code — Redux is now reserved mostly for large existing codebases or teams that specifically need its strict patterns.

Conceptual · Boilerplate Comparison 
// Redux: action type + action creator + reducer case — just to add 1
// Zustand: 
const useStore = create((set) => ({
  count: 0,
  increment: () => set((s) => ({ count: s.count + 1 })),
}));

What is TanStack Query (React Query) and what problem does it solve?

📖 Answer

TanStack Query is a library specialized in managing server/async state — data that lives on a backend and needs fetching, caching, and re-fetching. Instead of manually writing useState + useEffect + loading flags + error handling for every API call, you describe what data you need and TanStack Query handles caching, background refetching, retries, and deduplication automatically. Think of it as hiring a personal assistant for your data fetching — you ask for the latest report, and it figures out whether to grab a fresh copy or hand you the one it already has cached.

Bash · Install 
npm install @tanstack/react-query
# Wrap your app in <QueryClientProvider client={queryClient}>
78

What is the difference between useQuery and useMutation?

📖 Answer

useQuery is for reading data — GET requests like fetching a list of posts — and runs automatically as soon as the component mounts, with built-in caching. useMutation is for writing data — POST/PUT/DELETE requests like submitting a form — and only runs when you explicitly call its mutate() function, since you don't want a "create user" request firing just because a component rendered. Think of useQuery as checking your mailbox automatically every time you walk by, and useMutation as deciding yourself when to drop a letter in the mailbox to send.

JSX · useQuery vs useMutation 
// Reading — runs automatically
const { data, isLoading } = useQuery({
  queryKey: ['posts'],
  queryFn: fetchPosts,
});

// Writing — runs only when called
const { mutate } = useMutation({ mutationFn: createPost });
mutate({ title: 'New Post' });
79

What is "stale-while-revalidate" caching in TanStack Query?

📖 Answer

When you re-visit a screen, TanStack Query immediately shows the data already cached from last time (even though it might be slightly outdated, or "stale") so the UI feels instant — then quietly fetches a fresh copy in the background and swaps it in once it arrives. The user never sees a blank loading spinner for data they've already seen recently. It's like glancing at yesterday's weather forecast pinned on the fridge while a fresh one downloads in the background — you get something useful instantly, then it gets corrected moments later if needed.

JS · staleTime Config 
useQuery({
  queryKey: ['profile'],
  queryFn: fetchProfile,
  staleTime: 60 * 1000, // data considered "fresh" for 60s; no refetch needed
});
// After 60s, the cached data is shown instantly while a refetch happens silently
80

How does TanStack Query expose loading, error, and success states?

📖 Answer

useQuery returns an object with boolean flags you can branch your UI on directly: isLoading (true only during the very first fetch with no cached data yet), isError (the request failed, with details in error), and data (populated once the fetch succeeds). There's also isFetching, which is true during any fetch including background refetches, even if cached data is already showing. This removes the need to manually track three separate useState flags yourself for every single API call.

JSX · Handling States 
const { data, isLoading, isError, error } = useQuery({
  queryKey: ['todos'],
  queryFn: fetchTodos,
});

if (isLoading) return <ActivityIndicator />;
if (isError) return <Text>Error: {error.message}</Text>;
return <Text>Loaded {data.length} todos</Text>;
81

What is NativeWind and how does it bring Tailwind to React Native?

📖 Answer

NativeWind lets you style React Native components using Tailwind CSS's familiar utility classes — directly in a className string — instead of writing StyleSheet objects. Behind the scenes, it compiles those class names down into the same native style objects React Native would normally need; nothing actually runs as "CSS" on the device. It's like writing shorthand notes that get fully expanded into formal sentences before anyone reads them — you write less, but the underlying mechanism is identical to standard RN styling.

JSX · NativeWind className 
import { View, Text } from 'react-native';

<View className="flex-1 items-center justify-center bg-white">
  <Text className="text-xl font-bold text-indigo-600">Hello Tailwind!</Text>
</View>
82

What is the practical difference between using NativeWind and plain StyleSheet.create()?

📖 Answer

Functionally, they end up producing the same native style objects — NativeWind is essentially a build-time translator, not a new styling engine. The practical differences are developer experience: NativeWind classes are terser and colocated directly on the element (no separate style object to scroll to), and you get a consistent design scale (spacing, color shades) "for free" out of Tailwind's defaults. The trade-off is an extra build step and a learning curve if you don't already know Tailwind's utility-class naming conventions.

Conceptual · Same Output, Different Syntax 
// NativeWind
<View className="p-4 rounded-lg bg-indigo-600" />

// Equivalent plain StyleSheet
<View style={{ padding: 16, borderRadius: 8, backgroundColor: '#4338ca' }} />
83

What is Tamagui, and what makes it different from a typical styling library?

📖 Answer

Tamagui is a full UI kit and styling system that compiles your styles at build time (rather than at runtime) for better performance, and ships pre-built, themeable, accessible components (buttons, inputs, stacks) that work identically on iOS, Android, and Web from one codebase. Beyond just "styling," it provides an entire design-system layer — themes, responsive variants, and animation support baked in. Think of it less like a paint color (basic styling) and more like a fully-stocked architectural toolkit with pre-fabricated, customizable rooms ready to assemble.

JSX · Tamagui Stack & Button 
import { YStack, Button, Text } from 'tamagui';

<YStack space="$4" padding="$4" alignItems="center">
  <Text fontSize="$6">Welcome</Text>
  <Button theme="active">Get Started</Button>
</YStack>

What is Gluestack UI?

📖 Answer

Gluestack UI is a cross-platform, accessible component library (buttons, modals, form inputs, and more) built on a "copy the code into your project" model similar in spirit to shadcn/ui on the web — rather than installing components as an opaque npm dependency, you own the actual component source files and can freely modify them. It's commonly paired with utility-style theming (often NativeWind/Tailwind-flavored) for visual customization. Think of it as buying unassembled furniture you can rearrange however you like, rather than a sealed pre-built cabinet you can only use as-is.

JSX · Gluestack Button 
import { Button, ButtonText } from '@/components/ui/button';

<Button size="md" variant="solid" action="primary">
  <ButtonText>Continue</ButtonText>
</Button>
85

What are "design tokens" in modern React Native styling libraries?

📖 Answer

Design tokens are named, reusable values — like $space.4 for spacing or $color.primary for a brand color — that stand in for raw numbers and hex codes throughout your app. Instead of typing 16 or #4338ca everywhere, you reference the token; changing the token's definition in one place updates every component using it instantly. It's exactly like a paint company's color names ("Midnight Indigo") — painters across a thousand houses all use the same can, so repainting the brand's signature color everywhere means mixing one new batch, not visiting every house individually.

JS · Token Definition & Usage 
// Defined once in a theme/tokens file
const tokens = { color: { primary: '#4338ca' }, space: { 4: 16 } };

// Used everywhere via the library's token syntax
<View backgroundColor="$primary" padding="$4" />
// Change the token once → updates every screen using it
86

What is a "universal" component library, and why does it matter for RN + Web?

📖 Answer

A universal library (Tamagui and NativeWind both fit this label) ships components that render correctly on iOS, Android, and web from the exact same source code and the exact same import — no separate ".web.js" file or platform branching needed for basic styling. This matters because more 2026 teams are shipping a single React codebase to mobile and a marketing or admin website, and maintaining two separate design systems is expensive. It's like one universal power adapter that works in any country's outlet, rather than packing a different plug for every destination.

Conceptual · One Codebase, 3 Platforms 
// Same import, same JSX — runs unmodified on iOS, Android, and Web
import { Button } from 'tamagui';

export default function CTA() {
  return <Button>Sign Up</Button>;
}
87

What's the difference between a "styled-components" approach and a utility-class approach in RN styling?

📖 Answer

A styled-components-style approach defines a brand-new, named component with its styles baked in once (e.g., a StyledButton), which you then reuse everywhere as a single tag — styles live separately from markup. A utility-class approach (like NativeWind) applies many small, single-purpose classes directly inline on the existing element (p-4 rounded-lg bg-blue-500) — styles live right alongside the markup. It's the difference between ordering a pre-named combo meal once (styled-components) versus picking each individual item à la carte every time you order (utility classes).

Conceptual · Two Approaches 
// Styled-components style — define once, reuse as a tag
const StyledButton = styled(Pressable)\`padding: 16px; background: #4338ca;\`;
<StyledButton />

// Utility-class style — composed inline, every time
<Pressable className="p-4 bg-indigo-700" />
88

How do modern styling libraries typically handle dark mode and theming?

📖 Answer

Most modern libraries define two (or more) complete sets of token values — a "light" theme and a "dark" theme — that map the same token names (like $background or $text) to different actual colors. Switching themes is then just flipping a single active-theme flag; every component referencing tokens updates automatically without touching individual component code. Libraries typically also detect the device's system-level color scheme automatically via a hook. It's like a stage with colored gel filters over the lights — swap the filter, and every prop on stage instantly looks different, without anyone moving a single object.

JSX · Theme Switching 
import { useColorScheme } from 'react-native';

const scheme = useColorScheme(); // 'light' | 'dark', from system settings

<TamaguiProvider defaultTheme={scheme}>
  <App />
</TamaguiProvider>

What is the component composition / "slots" pattern used in libraries like Tamagui or Gluestack UI?

📖 Answer

Instead of one giant component with dozens of props to configure every internal piece, these libraries break a UI element into several small sub-components you assemble yourself — like Button, ButtonText, and ButtonIcon nested together. Each "slot" handles one part, and you only include the slots you actually need. It's like a modular sofa system — you don't buy one fixed-shape couch, you snap together exactly the cushions and arms you want, in the arrangement that fits your room.

JSX · Composable Slots 
import { Button, ButtonText, ButtonIcon } from '@/components/ui/button';
import { ArrowRight } from 'lucide-react-native';

<Button>
  <ButtonText>Next</ButtonText>
  <ButtonIcon as={ArrowRight} />
</Button>

As a beginner in 2026, which styling approach should you start with, and why?

📖 Answer

Start with plain StyleSheet.create() first — it forces you to actually understand Flexbox and React Native's style properties without any abstraction hiding what's happening underneath. Once that's solid, NativeWind is a natural next step if you already know (or want to learn) Tailwind's class-naming conventions, since it speeds up everyday styling significantly. Reach for Tamagui or Gluestack UI only once you're building a real multi-screen app and want consistent, pre-built, accessible components rather than building every button and input from scratch — they're powerful, but layering them on too early can hide the fundamentals you still need to learn.

Learning Path Guide 
1️⃣  StyleSheet.create()   → learn Flexbox & core RN style props first
2️⃣  NativeWind             → faster utility-class styling once fundamentals are solid
3️⃣  Tamagui / Gluestack UI → pre-built, themeable component systems for real apps
