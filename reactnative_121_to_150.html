Module 5 · Push Notifications via Firebase Cloud Messaging

This final module covers how apps reach users even when they're not actively looking at the screen. You'll learn the mechanics of Firebase Cloud Messaging (FCM) — device tokens and the three app states a notification can arrive in — how the native credentials and certificates behind the scenes actually get set up, and how to display notifications locally using expo-notifications or the more advanced Notifee library. This is a frequently-tested interview area because it touches native platform quirks that trip up many beginners.

121

What is Firebase Cloud Messaging (FCM) and what problem does it solve?

📖 Answer

FCM is Google's free, cross-platform messaging service that lets a backend server send a message to a specific app installed on a specific device, even if that app isn't currently open. It's the delivery infrastructure underneath nearly every push notification you've ever received — a backend doesn't talk directly to a phone; it hands the message to FCM, and FCM routes it through Apple's or Google's own push delivery networks to the exact device. Think of it as a postal service: your server writes the letter, but FCM is the courier network that actually knows how to reach the recipient's mailbox, wherever they are.

Conceptual · Delivery Path 
Your Backend Server
        ↓  sends message + device token
Firebase Cloud Messaging (FCM)
        ↓  routes via platform push network
Apple Push Notification service (APNs)  /  Google Play Services
        ↓
User's Device — notification appears 🔔

What is a push notification "token" or device token?

📖 Answer

A device token is a unique, long string generated by the push service that identifies one specific app installation on one specific device — essentially a mailing address that only your backend can use to deliver a message to that exact device. Without it, FCM has no way to know which of billions of devices to send a notification to. It's like a private PO box number: your server keeps a list of every user's box number, and sending mail simply means addressing it correctly to that box.

JSON · Example Token Shape 
// A real token is much longer — this is illustrative
"ExponentPushToken[xxxxxxxxxxxxxxxxxxxxxx]"

// Or raw FCM token format:
"fGq3K9z...:APA91bH..." // 100+ characters long
123

How is a device token generated and registered in a React Native app?

📖 Answer

When your app first requests notification permission and the user grants it, the device registers itself with the platform's push service (APNs on iOS, FCM directly on Android), which generates a unique token and hands it back to your app code. Your app's job is then to send that token to your own backend — typically saving it against the logged-in user's account in your database — so your server knows where to deliver future messages. The token itself is generated by the OS/push service; your app and backend are just responsible for capturing and storing it.

JS · Getting & Sending the Token 
import * as Notifications from 'expo-notifications';

const { data: token } = await Notifications.getExpoPushTokenAsync();

// Send it to your own backend to save against the user's account
await fetch('https://api.myapp.com/save-token', {
  method: 'POST',
  body: JSON.stringify({ userId, token }),
});

What is the difference between raw FCM and Expo's Push Notification service?

📖 Answer

Raw FCM requires your backend to talk directly to Google's servers (and separately to Apple's APNs for iOS), managing two different credential systems and message formats. Expo's Push Notification service sits as a friendly middle layer in front of both — your backend sends one simple request to Expo's servers using an Expo push token, and Expo handles translating and routing it to FCM or APNs correctly behind the scenes. It's like using one universal shipping company that handles customs paperwork for both international destinations, instead of dealing with two separate customs offices yourself.

Conceptual · One Layer vs Two 
Raw FCM:    Backend → FCM (Android) AND Backend → APNs (iOS) — 2 systems
Expo Push:  Backend → Expo Push API → (Expo routes to FCM/APNs for you)
125

What happens to a push notification when the app is in the Foreground?

📖 Answer

When the app is open and actively visible, the OS generally does not automatically show the system notification banner — instead, your JavaScript code receives the message directly through a listener, and it's entirely up to you whether and how to display something (a custom in-app banner, a toast, or nothing at all). This is intentional: since the user is already looking at your app, a jarring OS-level alert popping over their current screen would often be disruptive. Think of it as a phone call arriving while you're already mid-conversation with the caller in person — there's no need to also ring the doorbell.

JS · Foreground Listener 
import * as Notifications from 'expo-notifications';

Notifications.addNotificationReceivedListener((notification) => {
  // App is open right now — decide what to do yourself
  console.log('Received while foregrounded:', notification.request.content.title);
});
126

What happens to a push notification when the app is in the Background?

📖 Answer

When the app is minimized but still running in memory (the user switched to another app or the home screen), a standard notification message is handled entirely by the OS — it shows the native banner/alert in the system tray automatically, without your JavaScript code needing to do anything. Your app's code only "wakes up" again if the user actually taps the notification, which typically opens the app and fires a "response received" event you can listen for to navigate somewhere specific.

JS · Handling the Tap 
Notifications.addNotificationResponseReceivedListener((response) => {
  // Fires when the user TAPS the system notification
  const screen = response.notification.request.content.data?.screen;
  navigation.navigate(screen);
});
127

What happens when the app is Quit/Killed (fully terminated)?

📖 Answer

If the app process has been fully terminated (not just backgrounded), a standard notification still shows up via the OS, exactly like the background case — but your JavaScript runtime isn't running at all, so no listeners can fire until the app is actually launched. If the user taps the notification, the OS launches the app fresh, and you typically need to check for an "initial notification" on startup to know the app was opened because of a tap, rather than a normal manual launch.

JS · Checking Launch Source 
// On cold start — was the app launched BY tapping a notification?
const response = await Notifications.getLastNotificationResponseAsync();

if (response) {
  const screen = response.notification.request.content.data?.screen;
  // Navigate there once the app finishes mounting
}
128

What is the difference between a "data" message and a "notification" message in FCM?

📖 Answer

A notification message includes a title and body that the OS automatically displays in the system tray — your app doesn't have to do anything for it to appear. A data message contains only a custom key-value payload with no automatic visual — your app's JavaScript code must wake up and decide entirely on its own what to do with it (which might be nothing visible at all, like silently syncing fresh content). It's the difference between a sealed envelope with a visible "URGENT" stamp the mail carrier shows you immediately (notification) versus a plain unmarked envelope you have to personally open to know what's inside (data).

JSON · Two Payload Shapes 
// Notification message — OS shows this automatically
{ "notification": { "title": "New Message", "body": "Hi there!" } }

// Data-only message — app code decides what (if anything) to show
{ "data": { "type": "sync", "userId": "42" } }

How do you send a test push notification to a specific device token?

📖 Answer

If using Expo's push service, you can send a quick test directly via a simple HTTP POST to Expo's push API with the target token, title, and body — no backend code required for a manual test. This is extremely useful during development to confirm your token registration and notification display logic both actually work, before wiring up any real backend trigger logic.

Bash · curl Test Send 
curl -X POST https://exp.host/--/api/v2/push/send \
  -H "Content-Type: application/json" \
  -d '{
    "to": "ExponentPushToken[xxxxxxxxxxxx]",
    "title": "Hello!",
    "body": "This is a test notification."
  }'
130

Why can device tokens change or expire, and how should apps handle that?

📖 Answer

Tokens aren't permanent — they can rotate when the app is reinstalled, the user restores a device from backup, the underlying push service refreshes its credentials, or after long periods of inactivity. If your backend keeps sending to a stale token, delivery just silently fails. The fix is to register a listener for token refresh events and immediately re-send the new token to your backend whenever it fires, treating the token less like a permanent ID and more like a lease that needs periodic renewal.

Conceptual · Token Refresh Handling 
// Pseudo-flow — always keep your backend's copy current
onTokenRefresh((newToken) => {
  saveTokenToBackend(userId, newToken); // overwrite the old one
});
// Never assume a token captured once will remain valid forever
131

What is a Firebase project, and how do you connect it to a React Native app?

📖 Answer

A Firebase project is a container in Google's Firebase console that groups together all the backend services (Cloud Messaging, Analytics, Auth, etc.) for one app. Connecting it means registering your app's package name (Android) and bundle identifier (iOS) inside that project, which generates the two configuration files your native build needs. It's like opening a single account with a utility company that provides several services (electricity, water, gas) under one umbrella — registering your "address" (app identifiers) with that account is what activates the services for your specific building.

Conceptual · Connection Steps 
1. Create a project at console.firebase.google.com
2. Register your Android package name + iOS bundle ID inside it
3. Download google-services.json (Android) + GoogleService-Info.plist (iOS)
4. Add both files to your project / app.json config
132

What is google-services.json and where does it go?

📖 Answer

google-services.json is the Android configuration file you download from your Firebase project — it contains the project ID, API keys, and identifiers that let your Android app authenticate with Firebase services, including FCM. In a Managed Expo app, you reference its file path inside app.json under the Android config, and EAS Build automatically picks it up and embeds it correctly during the native build process — you never need to manually touch native Android folders.

JSON · app.json Reference 
{
  "expo": {
    "android": {
      "googleServicesFile": "./google-services.json"
    }
  }
}
133

What is GoogleService-Info.plist and where does it go?

📖 Answer

GoogleService-Info.plist is the iOS equivalent of google-services.json — a configuration file downloaded from your Firebase project containing the keys and identifiers your iOS app needs to talk to Firebase. Just like the Android file, in a Managed Expo project you simply reference its path inside app.json under the iOS config, and EAS Build embeds it into the native iOS project automatically during the build.

JSON · app.json Reference 
{
  "expo": {
    "ios": {
      "googleServicesFile": "./GoogleService-Info.plist"
    }
  }
}

What is an APNs certificate/key, and why does iOS need it in addition to Firebase?

📖 Answer

Apple requires its own authentication mechanism — an APNs (Apple Push Notification service) key or certificate, generated in your Apple Developer account — before Apple's push servers will deliver any message to an iPhone, regardless of which third-party service (Firebase included) is involved. Firebase actually uses your APNs key/certificate internally to relay messages through Apple's network on your behalf — it doesn't bypass Apple's own gatekeeping. It's like needing a customs stamp valid specifically for one country, even if a shared shipping company is handling the rest of the delivery route for both countries.

Conceptual · Two-Layer iOS Requirement 
iOS push delivery requires BOTH:
  1. An APNs Key/Cert from Apple Developer (uploaded into Firebase)
  2. Firebase's own project config (GoogleService-Info.plist)
// EAS credentials manager can generate/upload the APNs key for you
135

What is an FCM Server Key / Service Account, and why must it stay secret?

📖 Answer

The Server Key (or newer Service Account JSON file) is the credential that authorizes your backend to send messages through FCM on behalf of your Firebase project. Unlike the client-side config files (which are safe to bundle into your app), this key must never be embedded in your mobile app or committed to a public repository — anyone holding it could send unlimited spoofed notifications to all your users. It belongs exclusively on your server, kept as a protected secret, the same way you'd never tape your house key to your front door.

Conceptual · Where Each Key Belongs 
✅ Client app (bundled, OK to ship):
   google-services.json, GoogleService-Info.plist

🔒 Backend server ONLY (never ship to client, never commit publicly):
   FCM Service Account JSON / Server Key

How do EAS Build credentials work for push notifications in a Managed Expo app?

📖 Answer

EAS (Expo Application Services) can automatically generate, store, and manage the native signing credentials and push certificates your app needs — including the iOS APNs key — without you manually logging into Apple's developer portal yourself. Running eas credentials walks you through generating or uploading what's needed, and EAS securely stores it for future builds. This removes one of the most error-prone, manual parts of native push setup, which is exactly the kind of friction Expo's Managed workflow exists to eliminate.

Bash · EAS Credentials

137

What permission must be requested before notifications can be shown, on both iOS and modern Android?

📖 Answer

iOS has always required an explicit runtime permission prompt before any notification can be displayed — handled via requestPermissionsAsync() in expo-notifications. Android historically allowed notifications by default, but starting with Android 13, it now requires the same kind of explicit runtime permission request as iOS, closing a long-standing platform gap. Bottom line for 2026: always request notification permission explicitly on both platforms — never assume it's automatically granted.

JS · Requesting Permission 
import * as Notifications from 'expo-notifications';

const { status } = await Notifications.requestPermissionsAsync();
if (status !== 'granted') {
  console.log('Notifications will not be shown');
}

What is a Notification Channel on Android, and why is it required?

📖 Answer

Since Android 8 (Oreo), every notification must belong to a channel — a named category (e.g., "Messages," "Promotions") with its own importance level, sound, and vibration settings that the user can individually control from system settings. Without creating at least a "default" channel, notifications on modern Android either fail to display or fall back to the lowest-priority behavior. It's like a building with separate mailboxes for "Bills," "Personal Mail," and "Junk" — residents can choose to get an alert bell only for certain boxes, rather than an all-or-nothing setting for every letter.

JS · Creating a Channel 
await Notifications.setNotificationChannelAsync('default', {
  name: 'Default',
  importance: Notifications.AndroidImportance.HIGH,
  vibrationPattern: [0, 250, 250, 250],
});

What is the "Sender ID" in FCM?

📖 Answer

The Sender ID (also called the Project Number) is a unique numeric identifier for your Firebase project, used internally during the registration process so the device's push service knows which Firebase project is allowed to send it messages. It's automatically embedded inside your google-services.json file — as a beginner you rarely need to type it manually, but recognizing the term helps when reading Firebase documentation or debugging registration issues referencing it.

JSON · Found Inside google-services.json 
{
  "project_info": {
    "project_number": "123456789012"   // ← this is the Sender ID
  }
}

Why can't you fully test push notifications inside Expo Go, and what should you use instead?

📖 Answer

Expo Go is a generic pre-built app used for quick development previews, but it ships with its own fixed bundle identifier and certificates — it cannot be configured with your specific Firebase project's credentials or APNs keys. This means real, custom push notification setups (your own FCM project, your own iOS push certificates) simply don't work properly inside it. To test push notifications fully, you need a development build created via EAS Build, which compiles a real native app embedding your actual credentials — a one-time extra step that's now considered standard practice for any app needing native modules beyond what Expo Go ships with.

Bash · Building a Dev Client 
eas build --profile development --platform ios
# Installs a custom dev build with YOUR credentials baked in —
# this is what lets push notifications actually be testable

What is expo-notifications and what does it provide?

📖 Answer

expo-notifications is the official Expo SDK module for everything notification-related — requesting permission, fetching the push token, scheduling local notifications, listening for incoming messages, and configuring Android channels — all through one consistent JavaScript API that works across both platforms. It's the natural starting point for most apps' push notification needs before reaching for a more specialized library like Notifee for advanced custom UI.

Bash · Install 
npx expo install expo-notifications
import * as Notifications from 'expo-notifications';

How do you request notification permissions using expo-notifications, step by step?

📖 Answer

First check the existing status with getPermissionsAsync() (no prompt shown) — if it's not already 'granted', then call requestPermissionsAsync(), which triggers the native OS dialog. Always check before requesting, since repeatedly calling the request function on a device that already answered can be a no-op or, worse, confuse your permission-handling logic. This two-step pattern (check, then request only if needed) is the standard, safe approach used across all permission types in React Native, not just notifications.

JS · Check-Then-Request Pattern 
const existing = await Notifications.getPermissionsAsync();
let finalStatus = existing.status;

if (finalStatus !== 'granted') {
  const requested = await Notifications.requestPermissionsAsync();
  finalStatus = requested.status;
}

if (finalStatus === 'granted') {
  const token = await Notifications.getExpoPushTokenAsync();
}

How do you schedule a local notification with expo-notifications?

📖 Answer

A local notification is one your app schedules itself, entirely on-device, with no backend or push token involved — useful for reminders, timers, or "your subscription expires soon" type alerts. You call scheduleNotificationAsync() with the content (title/body) and a trigger describing when to fire it (a delay in seconds, a specific date, or a repeating interval). It's the difference between a push notification (mail delivered from somewhere else) and a local notification (a sticky note you wrote yourself and set to pop up later).

JS · Scheduling a Local Alert 
await Notifications.scheduleNotificationAsync({
  content: { title: 'Reminder', body: 'Time to stretch!' },
  trigger: { seconds: 60 }, // fires 60 seconds from now
});

What is Notifee, and how does it differ from expo-notifications?

📖 Answer

Notifee is a more advanced, lower-level notification library focused entirely on rich local notification display — offering far deeper control than expo-notifications: custom action buttons embedded in the notification, expandable images, custom progress bars, grouped notifications, and finer-grained Android channel control. It's not a replacement for FCM itself (it doesn't manage push tokens or backend delivery) — it's specifically about making the notification that appears on screen look and behave exactly how you want, once a message has already arrived. Think of expo-notifications as the reliable basic toolkit, and Notifee as the specialist's toolkit for polished, custom notification UI.

JS · Notifee Display Call 
import notifee from '@notifee/react-native';

await notifee.displayNotification({
  title: 'Download complete',
  body: 'Your file is ready to view.',
  android: { channelId: 'default', pressAction: { id: 'default' } },
});

How do you handle a notification tap to navigate to a specific screen?

📖 Answer

You attach a navigation-target value (like a screen name or an ID) inside the notification's data payload when it's sent or scheduled. Then, inside your tap-response listener, you read that data back and call your navigation library's navigate function with it. The key is making sure your navigation reference (often a global ref set up at the app's root) is accessible from wherever your notification listener lives, since the listener typically sits outside the normal component tree.

JS · Data-Driven Navigation 
// Sending: attach where to navigate as data
await Notifications.scheduleNotificationAsync({
  content: { title: 'New order', data: { screen: 'OrderDetails', id: 42 } },
  trigger: null,
});

// Handling tap: read it back and navigate
Notifications.addNotificationResponseReceivedListener((res) => {
  const { screen, id } = res.notification.request.content.data;
  navigationRef.current?.navigate(screen, { id });
});

What's the difference between the foreground listener and the response listener in expo-notifications?

📖 Answer

addNotificationReceivedListener fires whenever a notification arrives while the app is in the foreground — regardless of whether the user ever interacts with it. addNotificationResponseReceivedListener fires only when the user actually taps on a notification (or one of its action buttons), whether the app was foregrounded, backgrounded, or had to be launched from quit. They answer two different questions: "a message just showed up" versus "the user just engaged with one" — and most apps need logic in both places for a complete experience.

JS · Two Listener Types 
// "It arrived" — foreground only
Notifications.addNotificationReceivedListener((n) => { /* show in-app toast */ });

// "User tapped it" — any app state
Notifications.addNotificationResponseReceivedListener((r) => { /* navigate */ });

How do you display a custom-styled notification with an image using Notifee?

📖 Answer

Notifee's displayNotification accepts platform-specific style objects — on Android, you can set a BigPictureStyle with an image URI to show an expanded image when the notification is pulled down; on iOS, you attach an attachments array. This kind of rich-media notification (think a delivery app showing a photo of your food order) goes well beyond what plain title/body notifications can offer, and is one of the main reasons teams reach for Notifee over the more basic expo-notifications display API.

JS · Image-Rich Notification 
await notifee.displayNotification({
  title: 'Order on its way!',
  body: 'Tap to track your delivery.',
  android: {
    channelId: 'default',
    style: { type: AndroidStyle.BIGPICTURE, picture: 'https://example.com/food.jpg' },
  },
});

What is a "silent" / background data notification, and what is it used for?

📖 Answer

A silent notification (a data-only FCM message with no visible title/body) wakes your app's background code briefly without showing anything to the user at all — commonly used to trigger a quiet background sync, refresh cached content, or update a badge count, so that the next time the user opens the app, everything already feels up to date. It's like a courier quietly restocking your fridge while you're out, rather than ringing the doorbell to tell you a delivery happened — the result (fresh content) is there, but there's no announcement.

JSON · Silent Data-Only Payload 
{
  "data": { "type": "background-sync" }
  // No "notification" field → nothing visible appears to the user
}

How do you badge the app icon with an unread count?

📖 Answer

The small red number that appears on an app's home screen icon is called the badge count, and expo-notifications exposes a simple function to set it directly: setBadgeCountAsync(number). You typically update this whenever your backend's unread count changes, or include a badge field directly in the push payload itself so the OS updates it automatically on delivery, without needing your app to even be running.

JS · Setting the Badge Count 
import * as Notifications from 'expo-notifications';

await Notifications.setBadgeCountAsync(5);   // shows "5" on the app icon
await Notifications.setBadgeCountAsync(0);   // clears it once read

What are best practices for not annoying users with too many push notifications?

📖 Answer

Always request permission contextually (right before a feature that genuinely needs it, not on app launch), give users granular in-app controls over which notification categories they receive (using Android channels to back this), avoid sending multiple separate notifications for closely-related events (group or batch them instead), and never send notifications purely for re-engagement marketing without real value to the user. Apps with high "notifications disabled" or uninstall rates very often trace back to notification fatigue — interviewers asking this question are checking that you think about UX, not just the technical mechanics of sending alerts.

Best Practices Checklist 
✅ Ask permission just-in-time, with a clear, contextual reason
✅ Let users toggle categories (channels) on/off individually
✅ Batch/group related alerts instead of firing several at once
✅ Every notification should carry genuine value — not just "come back!"
✅ Respect quiet hours / Do Not Disturb where relevant
