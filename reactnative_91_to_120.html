Module 4 · Device Hardware: GPS & Maps

This module covers how React Native apps tap into real device hardware — the user's physical location. You'll learn how runtime permissions work (and why they're stricter than ever on both platforms), how to fetch and continuously track coordinates with expo-location, and how to render an interactive map with custom markers and tracked paths using react-native-maps. Location features are a favorite interview topic because they touch permissions, async APIs, and native UI all at once.

91

Why do mobile apps need to request runtime permissions for location?

📖 Answer

A user's GPS coordinates are extremely sensitive — they can reveal home address, daily routine, and real-time movements. Both iOS and Android enforce a runtime permission system: the operating system itself shows a native popup asking the user to explicitly allow or deny access, and your app code cannot read location data until that's granted. Think of it as a locked door inside the user's house — your app can knock and ask, but the OS (the homeowner) decides whether to let you in, every time, not just once at install.

Conceptual · Permission Flow 
App code calls: requestForegroundPermissionsAsync()
        ↓
OS shows native popup: "Allow [App] to access your location?"
        ↓
User taps Allow / Deny  →  result returned to your JS code

What is the difference between foreground and background location permission?

📖 Answer

Foreground permission only allows location access while your app is actively open and visible on screen — the moment the user switches away or locks their phone, access stops. Background permission allows your app to keep reading location even when minimized or closed, useful for fitness trackers logging a run or delivery apps tracking a driver. Background access is a much bigger privacy ask, so both platforms require it to be requested as a clearly separate, additional step — like the difference between a guest allowed in your living room while you're home (foreground) versus given a house key to come and go anytime (background).

JS · expo-location 
import * as Location from 'expo-location';

// Foreground only — app must be open
await Location.requestForegroundPermissionsAsync();

// Background — needs foreground granted FIRST
await Location.requestBackgroundPermissionsAsync();

How do you request foreground location permission using expo-location?

📖 Answer

You call Location.requestForegroundPermissionsAsync(), which is an async function that triggers the native OS permission dialog and resolves with a status field (either 'granted' or 'denied'). Best practice is to call this right before you actually need the location — not on app launch — and to check the returned status before attempting to fetch coordinates, since calling a location method without permission will throw an error.

JS · Requesting Permission 
import * as Location from 'expo-location';

async function askPermission() {
  const { status } = await Location.requestForegroundPermissionsAsync();
  if (status !== 'granted') {
    console.log('Permission denied — cannot fetch location');
    return false;
  }
  return true;
}
94

What is the difference between requestForegroundPermissionsAsync and requestBackgroundPermissionsAsync?

📖 Answer

requestForegroundPermissionsAsync asks for the basic "while using the app" permission and is almost always called first. requestBackgroundPermissionsAsync asks for the elevated permission allowing tracking even when the app isn't on screen — but on both iOS and Android, you must have foreground permission already granted before background permission can even be requested; calling it first will simply fail or be ignored. Think of it as needing a basic visitor badge before you can apply for an after-hours building key — you can't skip straight to the higher-trust permission.

JS · Correct Order 
const fg = await Location.requestForegroundPermissionsAsync();
if (fg.status === 'granted') {
  // Only NOW can background be requested
  const bg = await Location.requestBackgroundPermissionsAsync();
}
95

What happens if a user denies a location permission request?

📖 Answer

The permission function resolves with status: 'denied' rather than throwing — your app keeps running normally, it just can't access location data. Calling a location-fetching function anyway (without checking status first) will typically throw a runtime error. Importantly, on most platforms, once denied, calling the request function again often won't re-show the native dialog — many OS versions silently deny repeat requests, requiring the user to manually re-enable the permission from system Settings instead. Good apps detect this and guide the user there with a clear explanation.

JS · Handling Denial 
const { status, canAskAgain } = await Location.requestForegroundPermissionsAsync();

if (status !== 'granted' && !canAskAgain) {
  // Dialog won't show again — direct user to Settings manually
  Alert.alert('Permission needed', 'Please enable location in Settings.');
}

How do you check the current permission status without prompting the user?

📖 Answer

Location.getForegroundPermissionsAsync() (and its background equivalent) silently reads the current permission state without ever showing a dialog — useful for deciding what UI to show when a screen loads, before you commit to actually requesting anything. This is the "check" counterpart to the "ask" function — like quietly glancing at whether a door is already unlocked before deciding whether you need to knock at all.

JS · Checking, Not Requesting 
const { status } = await Location.getForegroundPermissionsAsync();

if (status === 'granted') {
  // Already allowed — fetch location directly
} else {
  // Not yet granted — now show your own UI explaining why, then request
}

What is the difference between "When In Use" and "Always" location permission on iOS?

📖 Answer

These are iOS's specific names for foreground vs. background access. "When In Use" grants location only while the app is actively open on screen. "Always" additionally permits background tracking. Apple's permission dialog is notably strict: it often first offers only a one-time "Allow Once" or "While Using the App" choice, and only presents the "Always" upgrade option later, typically after the app has already requested foreground access and demonstrated a need — Apple deliberately makes the highest-trust option harder to obtain to protect user privacy.

Conceptual · iOS Permission Tiers 
iOS dialog options typically shown to the user:
  "Allow Once"             → one-time, foreground only
  "Allow While Using App"  → foreground (== "When In Use")
  "Always Allow"           → foreground + background (upgrade, asked later)

What configuration is needed in app.json to enable location permissions?

📖 Answer

In a Managed Expo project, you add the expo-location config plugin to app.json, including custom permission description strings that explain why your app needs location — these exact strings are what the user sees inside the native OS permission popup. iOS requires this description text or your app will be rejected from the App Store; Android handles most of this automatically but still benefits from a clear rationale. Skipping this step is a very common beginner mistake that causes permission requests to silently fail.

JSON · app.json plugin config 
{
  "expo": {
    "plugins": [
      ["expo-location", {
        "locationAlwaysAndWhenInUsePermission":
          "Allow $(PRODUCT_NAME) to use your location to show nearby stores."
      }]
    ]
  }
}

Why might background location require extra justification for app store approval?

📖 Answer

Both Apple's App Store and Google Play treat background location as a high-risk privacy feature and require developers to clearly justify it during the review process — explaining exactly what user-facing feature needs it (e.g., live ride tracking, fitness route logging). Apps that request background access without an obvious matching feature are routinely rejected or removed. It's like a landlord requiring a written explanation before handing out a spare key — casual or vague reasons ("just in case") won't be accepted; the use case must be concrete and clearly tied to something the user directly benefits from.

Conceptual · Store Review Checklist 
✅ Acceptable justifications:
   - Live navigation / turn-by-turn directions
   - Fitness app tracking a run/ride in progress
   - Delivery app tracking driver en route

❌ Likely rejected:
   - "Improve user experience" with no concrete feature shown

What is a good UX pattern for asking permissions: just-in-time vs. upfront?

📖 Answer

Best practice is just-in-time requesting: ask for a permission only at the exact moment the user takes an action that needs it (e.g., tapping "Find stores near me"), not immediately when the app launches. Users are far more likely to grant a permission when they understand exactly why it's needed in context, versus a vague popup on a blank welcome screen. It's like a waiter asking "Can I take your coat?" right as you sit down (clear, contextual) versus a stranger demanding your coat the second you walk through the door (confusing, off-putting) — context dramatically improves the odds of a "yes."

JSX · Just-In-Time Request 
// ❌ Bad — requesting on app launch, no context
useEffect(() => { Location.requestForegroundPermissionsAsync(); }, []);

// ✅ Good — requested only when the user taps a relevant button
<Pressable onPress={async () => {
  await Location.requestForegroundPermissionsAsync();
  findNearbyStores();
}}>
  <Text>Find stores near me</Text>
</Pressable>

What is expo-location and what does it provide?

📖 Answer

expo-location is the official Expo SDK module that wraps the device's native GPS and location services into a simple JavaScript API. It handles permission requests, one-time coordinate fetches, continuous tracking, geocoding (address ↔ coordinates conversion), and more — without you ever touching native Swift/Kotlin code. Think of it as a translator standing between your JavaScript and the phone's actual GPS chip, converting your simple function calls into the right native instructions and handing results back in a clean, consistent format.

Bash · Install 
npx expo install expo-location
# Then import it in your component:
import * as Location from 'expo-location';

How do you get the user's current location once with getCurrentPositionAsync?

📖 Answer

Location.getCurrentPositionAsync() is an async function that takes a single GPS reading right now and resolves with the coordinates — perfect for a "show my current location" button that only needs one snapshot, not continuous tracking. You should always confirm permission is granted before calling it; otherwise it rejects with an error. It's like taking a single photograph of where you're standing right now, rather than recording a video that follows you as you move.

JS · One-Time Location Fetch 
const { status } = await Location.requestForegroundPermissionsAsync();
if (status === 'granted') {
  const location = await Location.getCurrentPositionAsync({});
  console.log(location.coords.latitude, location.coords.longitude);
}
103

What does the Location object returned by expo-location actually contain?

📖 Answer

The resolved object has a coords property holding latitude and longitude (the core position), plus altitude, accuracy (the GPS reading's margin of error in meters), heading (compass direction of travel), and speed (movement speed in meters/second). There's also a top-level timestamp marking exactly when the reading was taken. The accuracy field is especially important to check — a reading with a 500-meter accuracy radius is far less trustworthy than one with 5 meters.

JSON · Location Object Shape 
{
  coords: {
    latitude: 33.5651, longitude: 73.0169,
    altitude: 487.2, accuracy: 8.5,
    heading: 142.3,  speed: 0
  },
  timestamp: 1782460800000
}
104

What is the difference between getCurrentPositionAsync and watchPositionAsync?

📖 Answer

getCurrentPositionAsync takes exactly one reading and resolves — done. watchPositionAsync sets up continuous tracking: you give it a callback function, and it keeps calling that callback with a fresh coordinate every time the device moves (or a time interval elapses), until you explicitly stop it. Use the first for a one-off "where am I now" button, and the second for live tracking — like the difference between checking your speedometer once versus watching it constantly update as you drive.

JS · One-Shot vs Continuous 
// One-time snapshot
const loc = await Location.getCurrentPositionAsync({});

// Continuous — callback fires repeatedly as the device moves
const subscription = await Location.watchPositionAsync(
  { accuracy: Location.Accuracy.High, distanceInterval: 10 },
  (newLocation) => console.log(newLocation.coords)
);
105

What does the "accuracy" option control when fetching location?

📖 Answer

The accuracy option tells the device how precise a reading you need, which directly trades off against battery usage and speed. Location.Accuracy.Lowest might use cell towers or Wi-Fi (fast, low battery cost, accurate to ~kilometers) while Location.Accuracy.BestForNavigation uses full GPS hardware continuously (slower to acquire, drains battery faster, accurate to a few meters). Choosing the right level is like deciding between a quick glance at a city map (low accuracy, low effort) versus carefully reading street-level GPS coordinates (high accuracy, more effort) — pick only as much precision as your feature actually needs.

JS · Accuracy Levels 
await Location.getCurrentPositionAsync({
  accuracy: Location.Accuracy.High, // good balance for most map features
});
// Other levels: Lowest, Low, Balanced, High, Highest, BestForNavigation
106

How do you continuously track a user's location with watchPositionAsync?

📖 Answer

You call watchPositionAsync with a configuration object (specifying accuracy and how often updates should fire, via timeInterval and/or distanceInterval) and a callback that receives each new location as it arrives. Typically you save each update into React state so your UI (like a map marker) re-renders to reflect the latest position. This is the foundation for any "live tracking" feature, like a runner's route or a delivery driver's progress on a map.

JSX · Live Tracking with State 
const [position, setPosition] = useState(null);

useEffect(() => {
  let sub;
  (async () => {
    sub = await Location.watchPositionAsync(
      { accuracy: Location.Accuracy.High, distanceInterval: 5 },
      (loc) => setPosition(loc.coords)
    );
  })();
  return () => sub?.remove();   // cleanup on unmount
}, []);
107

How do you stop watching location updates, and why is cleanup important?

📖 Answer

watchPositionAsync returns a subscription object with a .remove() method — calling it stops the GPS watch entirely. This cleanup is critical: if you forget to call it when a screen unmounts (e.g., the user navigates away), the GPS hardware keeps running silently in the background, draining battery for no visible benefit — a very common, easy-to-miss bug. Always call .remove() inside a useEffect cleanup function, the same way you'd turn off a faucet after filling a glass instead of letting it run indefinitely.

JS · Subscription Cleanup 
useEffect(() => {
  let subscription;
  Location.watchPositionAsync({}, (loc) => setPosition(loc.coords))
    .then((sub) => { subscription = sub; });

  return () => {
    subscription?.remove(); // ✅ stops GPS watch — saves battery
  };
}, []);

What is geocoding and reverse geocoding in expo-location?

📖 Answer

Geocoding converts a human-readable address into coordinates — "1600 Amphitheatre Parkway" becomes a latitude/longitude pair, via Location.geocodeAsync(). Reverse geocoding does the opposite: it takes coordinates and returns a readable address, via Location.reverseGeocodeAsync() — useful for showing "You are near Main Street" instead of raw numbers. Think of geocoding as looking up a street address in a phone book to find its map coordinates, and reverse geocoding as pointing at a spot on a map and asking "what's the address here?"

JS · Geocoding Both Ways 
// Address → coordinates
const results = await Location.geocodeAsync('Times Square, New York');

// Coordinates → address
const address = await Location.reverseGeocodeAsync({
  latitude: 40.7580, longitude: -73.9855,
});
console.log(address[0].city, address[0].street);

Why might location be slow or inaccurate indoors, and how does GPS differ from network-based location?

📖 Answer

True GPS relies on receiving signals from orbiting satellites, which requires a relatively clear line of sight to the sky — walls, roofs, and especially underground spaces block or weaken these signals, causing slow "time to first fix" or wildly inaccurate readings indoors. As a fallback, devices can use network-based location — triangulating position from nearby Wi-Fi networks and cell towers, which works indoors but is far less precise (tens to hundreds of meters, vs. GPS's few meters). It's the difference between using stars to navigate (GPS — precise outdoors, useless under a roof) versus recognizing nearby landmarks through a window (network-based — works anywhere, but blurrier).

Conceptual · GPS vs Network Location 
GPS (satellite)        → ~3-10m accuracy outdoors, poor indoors, slower fix
Network (Wi-Fi/cell)   → ~50-500m accuracy, works indoors, fast fix
// Most devices auto-blend both — that's what "Balanced" accuracy uses

How do you handle location errors gracefully, like GPS being turned off?

📖 Answer

Always wrap location calls in a try/catch — if location services are disabled at the OS level (not just app permission, but the device's master location toggle), getCurrentPositionAsync rejects with an error rather than hanging forever. You can also proactively check with Location.hasServicesEnabledAsync() before attempting a fetch, and show the user a friendly message guiding them to turn on location services in their device settings, rather than letting the app appear frozen or silently fail.

JS · Graceful Error Handling 
const servicesOn = await Location.hasServicesEnabledAsync();
if (!servicesOn) {
  Alert.alert('Location Off', 'Please enable Location Services in Settings.');
  return;
}

try {
  const loc = await Location.getCurrentPositionAsync({});
} catch (err) {
  console.log('Could not fetch location:', err.message);
}

What is react-native-maps and what does it render under the hood?

📖 Answer

react-native-maps is the standard library for embedding an interactive map inside a React Native screen. Rather than drawing its own map graphics, it wraps the device's native mapping engine — Apple Maps on iOS by default, and Google Maps on Android by default — giving you smooth, hardware-accelerated panning/zooming exactly like the platform's own Maps app, controlled entirely through React props and components.

JSX · Minimal MapView 
import MapView from 'react-native-maps';

<MapView
  style={{ flex: 1 }}
  initialRegion={{
    latitude: 33.5651, longitude: 73.0169,
    latitudeDelta: 0.05, longitudeDelta: 0.05,
  }}
/>
112

What is the region prop on MapView, and what do latitudeDelta/longitudeDelta control?

📖 Answer

The region (or initialRegion) prop defines exactly what part of the world the map shows — its center point (latitude/longitude) and its zoom level, expressed as latitudeDelta/longitudeDelta: the span of the map visible in degrees. A smaller delta means a more zoomed-in view (showing a few city blocks), while a larger delta zooms out (showing an entire country). Think of the center coordinates as where you're pointing a camera, and the deltas as how wide-angle versus zoomed-in the lens is.

JSX · Zoom via Deltas 
// Zoomed in — neighborhood-level view
{ latitude: 33.5651, longitude: 73.0169, latitudeDelta: 0.01, longitudeDelta: 0.01 }

// Zoomed out — whole-city view
{ latitude: 33.5651, longitude: 73.0169, latitudeDelta: 0.3,  longitudeDelta: 0.3  }

How do you add a basic marker to a map?

📖 Answer

You render a <Marker> component as a child of <MapView>, giving it a coordinate prop with the latitude/longitude where the pin should appear. You can render any number of markers, typically generated from an array of data (like store locations) using .map(). By default it renders the platform's standard pin-drop icon, with an optional title and description shown when tapped.

JSX · Basic Marker 
import MapView, { Marker } from 'react-native-maps';

<MapView style={{ flex: 1 }} initialRegion={region}>
  <Marker
    coordinate={{ latitude: 33.5651, longitude: 73.0169 }}
    title="My Office"
    description="Where I work every day"
  />
</MapView>

How do you create a custom marker with your own icon or image?

📖 Answer

Instead of relying on the default pin, you place any JSX as a child inside the <Marker> component — an <Image>, a styled <View>, even an emoji wrapped in <Text>. Whatever you render becomes the marker's visual appearance at that coordinate, completely replacing the default pin. It's like swapping a standard map pushpin for a custom-shaped sticker — the position logic stays the same, only the visual stamp changes.

JSX · Custom Marker Content 
<Marker coordinate={{ latitude: 33.5651, longitude: 73.0169 }}>
  <View style={{ backgroundColor: '#4338ca', padding: 6, borderRadius: 20 }}>
    <Text style={{ fontSize: 18 }}>🚗</Text>
  </View>

What is a Callout, and how do you show extra info when tapping a marker?

📖 Answer

A <Callout> is a small popup bubble that appears above a marker when the user taps it, similar to the speech-bubble info window you see in Google Maps. You nest it inside a <Marker> and put any custom JSX inside it — text, images, even buttons — for richer information than the basic title/description props allow. Think of it as a tooltip specifically designed for map pins: tap the pin, a little info card pops up right above it.

JSX · Custom Callout 
import { Marker, Callout } from 'react-native-maps';

<Marker coordinate={coords}>
  <Callout>
    <View style={{ padding: 8 }}>
      <Text style={{ fontWeight: 'bold' }}>Coffee Shop</Text>
      <Text>Open until 9 PM</Text>
    </View>
  </Callout>
</Marker>

How do you draw a path or route between coordinates using Polyline?

📖 Answer

<Polyline> draws a connected line across an ordered array of coordinates — give it a coordinates array (each with a latitude/longitude), and it draws straight segments connecting them in sequence, with customizable color and thickness via strokeColor and strokeWidth. This is exactly how apps draw a traveled route or a navigation path — each GPS reading collected over time becomes one point in the array, and the Polyline visually "connects the dots" as a continuous trail.

JSX · Polyline Path 
import { Polyline } from 'react-native-maps';

<Polyline
  coordinates={[
    { latitude: 33.5651, longitude: 73.0169 },
    { latitude: 33.5700, longitude: 73.0200 },
    { latitude: 33.5740, longitude: 73.0250 },
  ]}
  strokeColor="#4338ca"
  strokeWidth={4}
/>

How do you animate the map camera to a new region with animateToRegion?

📖 Answer

You call .animateToRegion(region, duration) on a ref attached to your <MapView> — this smoothly pans and zooms the camera to the new coordinates over the given duration in milliseconds, rather than jarringly snapping there instantly. It's commonly triggered by a "recenter on my location" button. Think of it as the difference between teleporting (jarring, disorienting) and a smooth camera pan in a movie (animateToRegion) — the gradual motion helps users keep their spatial bearings.

JSX · Camera Animation via Ref 
const mapRef = useRef(null);

function recenter(coords) {
  mapRef.current?.animateToRegion({
    latitude: coords.latitude,
    longitude: coords.longitude,
    latitudeDelta: 0.02, longitudeDelta: 0.02,
  }, 800); // animate over 800ms
}

<MapView ref={mapRef} style={{ flex: 1 }} initialRegion={region} />

How do you track and visually update a user's live position on the map?

📖 Answer

You combine watchPositionAsync with React state and a <Marker>: every time the watch callback fires with a new coordinate, you update state, which re-renders the Marker at its new coordinate prop — React Native handles smoothly moving the pin (and optionally re-centers the camera via animateToRegion). It's exactly how a ride-sharing app shows a car icon gliding across the map in near real-time — the icon doesn't actually "move" continuously; it's just redrawn at a fresh position dozens of times per minute.

JSX · Live-Updating Marker 
const [position, setPosition] = useState(null);

useEffect(() => {
  const sub = Location.watchPositionAsync({ distanceInterval: 5 },
    (loc) => setPosition(loc.coords));
  return () => { sub.then(s => s.remove()); };
}, []);

<MapView style={{ flex: 1 }} initialRegion={region}>
  {position && <Marker coordinate={position} />}
</MapView>

What is the difference between PROVIDER_GOOGLE and the default map provider?

📖 Answer

By default, react-native-maps uses each platform's native map engine — Apple Maps on iOS, Google Maps on Android. Passing provider={PROVIDER_GOOGLE} forces Google Maps to render on iOS too, giving a consistent look and feature set (like certain map styles or traffic layers) across both platforms instead of two visually different maps. The trade-off is needing a valid Google Maps API key configured for iOS, which Apple Maps doesn't require. It's like choosing to use the same brand of car on every road trip regardless of country, instead of renting whatever's locally standard.

JSX · Forcing Google Maps 
import MapView, { PROVIDER_GOOGLE } from 'react-native-maps';

<MapView
  provider={PROVIDER_GOOGLE}   // same engine on iOS AND Android
  style={{ flex: 1 }}
  initialRegion={region}
/>

What are common performance considerations when rendering many markers on a map?

📖 Answer

Rendering hundreds or thousands of <Marker> components at once — especially custom JSX markers rather than the default pin — can seriously slow down panning and zooming, since each one is a real native view. Common fixes include: using simpler default markers instead of complex custom JSX where possible, only rendering markers currently within the visible region (filtering by the map's bounds), and clustering nearby markers into a single "12 nearby" bubble at lower zoom levels (via libraries like react-native-map-clustering). It's like a city street map only showing a single "Downtown" label when zoomed way out, instead of trying to render every individual shop sign at once.

Conceptual · Marker Performance Tactics 
✅ Use the default pin instead of heavy custom Marker JSX when possible
✅ Filter markers to only those within the current visible region
✅ Cluster nearby markers at low zoom (react-native-map-clustering)
✅ Avoid re-rendering all markers on every minor region change (memoize)
