What is the "New Architecture" in React Native (Fabric + TurboModules)?

📖 Answer

The New Architecture is a from-the-ground-up rebuild of React Native's internals, now the default for new projects. Fabric is the new rendering system that draws your UI, replacing the old renderer with one that can work synchronously and supports React 18 features like Concurrent Rendering. TurboModules is the new system for native modules — they're now loaded lazily (only when actually used) instead of all upfront, and communicate more directly with JavaScript. Together, they remove the old performance bottleneck of constantly serializing data back and forth across a bridge.

Conceptual · Old vs New 
Old Architecture:  JS ⇄ Bridge (async, JSON serialized) ⇄ Native
New Architecture:  JS ⇄ JSI (direct, synchronous calls) ⇄ Native
                   Fabric (rendering) + TurboModules (native modules)

What is JSI (JavaScript Interface) and why does it matter?

📖 Answer

JSI is a lightweight C++ layer that lets JavaScript hold direct references to native objects and call native functions synchronously, without needing to serialize data into JSON and pass it across an asynchronous bridge. This is the foundational piece that makes both Fabric and TurboModules possible. It matters because it eliminates an entire category of performance problems and timing bugs that came from the old bridge's async, batched nature — like the difference between shouting a question across a courtyard and waiting for an echo back (the bridge) versus simply reaching out and touching the object directly (JSI).

Conceptual · Why JSI Matters 
Old Bridge: JS → JSON.stringify → native queue → JSON.parse → Native
            (always async, always serialized — adds latency)

JSI:        JS → direct C++ function call → Native
            (can be synchronous — no serialization overhead)

What is Hermes, and why is it React Native's recommended JavaScript engine?

📖 Answer

Hermes is an open-source JavaScript engine built by Meta specifically optimized for React Native — unlike general-purpose engines like JSC (JavaScriptCore) or V8, Hermes pre-compiles your JS into efficient bytecode ahead of time (during the build), rather than parsing and compiling raw JavaScript on every app launch. This results in faster app startup time, lower memory usage, and a smaller app size — all of which matter more on lower-end mobile hardware than on a desktop browser. It's been the default engine for new React Native apps since 2022.

Conceptual · Why Hermes Helps 
Other engines:  Parse JS → Compile → Run        (every app launch)
Hermes:         Pre-compile to bytecode at BUILD time → just Run
                ✅ Faster startup  ✅ Lower memory  ✅ Smaller app size

What was the "bridge" in the old architecture, and why was it a performance bottleneck?

📖 Answer

In the original React Native architecture, JavaScript and native code lived in two completely separate worlds that could only communicate by serializing messages into JSON, queuing them, and processing them asynchronously in batches — this channel was called "the bridge." Every interaction (a button tap, a layout update) meant a round trip through this serialization process, which added latency and made truly synchronous native calls impossible. It's like two offices that can only communicate via faxed memos instead of a phone call — workable, but slow and prone to messages arriving out of order or with delay.

Conceptual · The Old Bridge 
JS Thread          Bridge (async, JSON)          Native Thread
   |  ──── "update this view" (serialized) ────────►  |
   |  ◄──── response, eventually, in a batch ───────   |
// Every single interaction paid this round-trip cost

What is a Native Module, and when would you actually need to write one?

📖 Answer

A Native Module is code written in a platform's native language (Swift/Objective-C for iOS, Kotlin/Java for Android) that exposes functionality to your JavaScript code — used when you need to access a device capability that no existing JS library covers, like a specialized Bluetooth protocol or a proprietary SDK a third-party vendor only ships natively. As a beginner, you'll rarely write one yourself; the vast majority of common needs (camera, location, storage, notifications) already have a well-maintained Expo or community library wrapping the native code for you.

Conceptual · When You'd Need One 
✅ Use an existing library — covers ~95% of common needs
   (camera, location, storage, biometrics, notifications, etc.)

🛠️ Write a Native Module only when:
   - No JS library wraps the native SDK you need
   - You need direct access to a proprietary/specialized hardware API

What is the difference between the built-in Animated API and Reanimated?

📖 Answer

React Native's built-in Animated API runs most animation logic on the JavaScript thread by default (unless you opt into useNativeDriver: true, which only supports a limited set of properties). react-native-reanimated is a community library that runs animation logic directly on the UI thread via "worklets" — small JS functions compiled to run natively — enabling much more complex, fully native-thread animations (including ones that respond to gestures) without ever blocking on the JS thread. It's the difference between a stagehand manually moving props during a performance (Animated, JS-thread-bound) versus an automated rig that runs the cues itself without needing to check in (Reanimated, UI-thread).

JSX · Animated vs Reanimated 
// Built-in Animated — runs on JS thread unless useNativeDriver set
Animated.timing(opacity, { toValue: 1, useNativeDriver: true }).start();

// Reanimated — worklet runs on the UI thread directly
const opacity = useSharedValue(0);
opacity.value = withTiming(1);
157

What is a "worklet" in react-native-reanimated and why does it matter for performance?

📖 Answer

A worklet is a small, specially-marked JavaScript function that Reanimated compiles to run directly on the UI thread instead of the regular JS thread. Because animations defined this way never need to cross back and forth to the JS thread on every frame, they stay smooth even when the JS thread is busy doing other work (like a heavy network response being processed) — a problem that would normally cause visible animation stutter or "jank" in the default Animated API. It's like giving the orchestra's conductor (UI thread) the sheet music in advance, so the music keeps playing smoothly even if someone in the front office (JS thread) gets distracted by a phone call.

JS · Marking a Worklet 
const animatedStyle = useAnimatedStyle(() => {
  'worklet'; // marks this to run on the UI thread
  return { opacity: opacity.value };
});

What is react-native-gesture-handler and why is it preferred over the built-in PanResponder?

📖 Answer

react-native-gesture-handler is a library that recognizes touch gestures (taps, drags, pinches, swipes) using the native gesture recognition systems on each platform, rather than reconstructing gesture logic entirely in JavaScript like the older built-in PanResponder does. This makes gestures feel more responsive and "native," and it integrates directly with Reanimated to drive animations entirely off the JS thread. Most navigation libraries (including React Navigation's swipe-back gesture) and modern UI libraries now depend on it as a baseline requirement.

JSX · Basic Pan Gesture 
import { GestureDetector, Gesture } from 'react-native-gesture-handler';

const pan = Gesture.Pan().onUpdate((e) => {
  translateX.value = e.translationX;
});

<GestureDetector gesture={pan}>
  <Animated.View style={animatedStyle} />
</GestureDetector>

What is React.memo and when should you use it?

📖 Answer

React.memo wraps a component so React skips re-rendering it if its props haven't actually changed since the last render — even if its parent re-rendered. This is useful for components that render often as siblings of frequently-updating state, but themselves receive the same props most of the time (like a static row in a list whose parent re-renders for unrelated reasons). It's not free, though — the comparison check itself has a small cost, so wrapping every single component "just in case" can sometimes do more harm than good; reach for it when you've actually identified unnecessary re-renders.

JSX · React.memo 
const UserRow = React.memo(function UserRow({ name }) {
  console.log('Rendered:', name);
  return <Text>{name}</Text>;
});
// Skips re-render if `name` prop is the same as last time
160

What is useMemo and what problem does it solve?

📖 Answer

useMemo caches the result of an expensive calculation between re-renders, only recomputing it when one of its listed dependencies actually changes. Without it, an expensive function (like filtering or sorting a large array) would re-run on every single re-render, even ones triggered by completely unrelated state changes elsewhere in the component. It's like keeping yesterday's calculated total in a notebook instead of redoing the entire long addition from scratch every time someone glances at the page, only recalculating when one of the underlying numbers actually changes.

JSX · useMemo 
const sortedUsers = useMemo(() => {
  return users.slice().sort((a, b) => a.name.localeCompare(b.name));
}, [users]); // only re-sorts when `users` itself changes
161

What is useCallback and how is it different from useMemo?

📖 Answer

useCallback caches a function definition itself between re-renders, rather than a computed value — without it, every re-render creates a brand-new function reference, which matters when you pass that function as a prop to a React.memo-wrapped child, since a "new" function prop would defeat the memoization and force a re-render anyway. useMemo returns the result of calling a function; useCallback returns the function itself, unchanged, as long as its dependencies haven't changed. Think of useMemo as caching a finished cake, and useCallback as caching the recipe card itself so you're not handing out a freshly photocopied recipe every single time.

JSX · useCallback vs useMemo 
// useMemo — caches a VALUE
const total = useMemo(() => computeTotal(items), [items]);

// useCallback — caches a FUNCTION reference
const handlePress = useCallback(() => {
  console.log('pressed', id);
}, [id]);

What causes unnecessary re-renders in React Native, and how do you detect them?

📖 Answer

The most common causes are: a parent re-rendering and passing brand-new inline object/array/function props every time (even with identical values, a new reference still triggers a re-render of non-memoized children), overly broad Context values causing every consumer to update, and state stored higher in the tree than it needs to be. To detect them, you can temporarily add console.log statements inside suspect components, use React DevTools' Profiler tab to record a render and see exactly which components fired and why, or reach for a library like why-did-you-render for automatic flagging during development.

Conceptual · Common Culprits 
// ❌ New object/function reference on EVERY render
<Child style={{ padding: 10 }} onPress={() => doThing()} />

// ✅ Stable references via memoization
const style = useMemo(() => ({ padding: 10 }), []);
const onPress = useCallback(() => doThing(), []);
<Child style={style} onPress={onPress} />

What is AsyncStorage and what is it used for?

📖 Answer

AsyncStorage is a simple, unencrypted, persistent key-value storage system built into React Native — data saved with it survives app restarts (though not an app uninstall). It's used for non-sensitive data you want to persist locally, like a user's theme preference, an onboarding-completed flag, or a cached copy of recently viewed items. Every read and write is asynchronous (hence the name), since it goes through native disk I/O. For anything sensitive — passwords, auth tokens — use SecureStore instead, never plain AsyncStorage.

JS · AsyncStorage Basics 
import AsyncStorage from '@react-native-async-storage/async-storage';

await AsyncStorage.setItem('theme', 'dark');
const theme = await AsyncStorage.getItem('theme'); // "dark"
await AsyncStorage.removeItem('theme');

What is SecureStore (expo-secure-store), and how is it different from AsyncStorage?

📖 Answer

expo-secure-store saves key-value pairs using the device's native encrypted storage — the iOS Keychain and Android Keystore — rather than the plain, unencrypted storage AsyncStorage uses. It's specifically intended for sensitive data: authentication tokens, refresh tokens, API secrets, or anything you wouldn't want exposed if someone gained physical access to an unlocked device's file system. Think of AsyncStorage as a notebook left on a desk anyone nearby could flip open, and SecureStore as the same notebook locked inside a small safe built into the desk.

JS · SecureStore Basics 
import * as SecureStore from 'expo-secure-store';

await SecureStore.setItemAsync('authToken', 'abc123...');
const token = await SecureStore.getItemAsync('authToken');

How do you make a network request and handle loading/error states without a library?

📖 Answer

React Native ships with the standard web fetch() API built in, so no extra install is needed for basic requests. The manual pattern is: a state for the data, a boolean for loading, and a slot for any error — set loading true, attempt the fetch inside a try/catch, store the result or the error, then set loading false in a finally block so it always resolves regardless of outcome. This is exactly the boilerplate that libraries like TanStack Query exist to eliminate once an app has more than a couple of these calls.

JS · Manual fetch Pattern 
const [data, setData] = useState(null);
const [loading, setLoading] = useState(true);
const [error, setError] = useState(null);

useEffect(() => {
  (async () => {
    try {
      const res = await fetch('https://api.example.com/users');
      if (!res.ok) throw new Error('Request failed');
      setData(await res.json());
    } catch (e) {
      setError(e.message);
    } finally {
      setLoading(false);
    }
  })();
}, []);

What is the difference between fetch and axios in React Native?

📖 Answer

fetch is built in and requires no install, but it's fairly bare-bones — it doesn't reject on HTTP error status codes (you must manually check res.ok), and it requires more boilerplate for things like setting default headers or timeouts. axios is a popular third-party library that adds convenience: automatic JSON parsing, rejecting promises on error status codes, request/response interceptors (great for automatically attaching an auth token to every request), and built-in request cancellation/timeout support. For simple apps, plain fetch is fine; for apps with many endpoints and shared auth logic, axios's interceptors save a lot of repeated code.

JS · fetch vs axios 
// fetch — must manually check .ok
const res = await fetch(url);
if (!res.ok) throw new Error('Failed');
const data = await res.json();

// axios — rejects automatically on error, JSON parsed for you
const { data } = await axios.get(url);

What is Platform.OS, and how do you write platform-specific code with it?

📖 Answer

Platform.OS is a simple string value — 'ios' or 'android' — that tells your code which operating system it's currently running on, letting you branch logic or styling for platform-specific differences (like shadow styling, which works very differently between the two). There's also Platform.select(), which lets you pick an entire value based on platform in a single, cleaner expression rather than nested ternaries scattered everywhere.

JS · Platform.OS & select 
import { Platform } from 'react-native';

if (Platform.OS === 'ios') { /* iOS-only logic */ }

const shadow = Platform.select({
  ios: { shadowOpacity: 0.2, shadowRadius: 4 },
  android: { elevation: 4 },
});

What is the .ios.js / .android.js file extension convention?

📖 Answer

Instead of branching logic inside one file with Platform.OS checks, you can create two entirely separate files — Button.ios.js and Button.android.js — and Metro (React Native's bundler) automatically picks the correct one at build time based on the target platform, while everywhere else in your code you simply import Button from './Button' with no extension. This is the right tool when a component's implementation differs so much between platforms that maintaining one shared file with heavy branching would be messier than just keeping two clean, separate versions.

File Tree · Platform-Specific Files 
components/
├── Button.ios.js       ← used automatically on iOS builds
└── Button.android.js   ← used automatically on Android builds

// Anywhere else:
import Button from './components/Button'; // no extension needed

What is SafeAreaView and why is it needed?

📖 Answer

Many phones have notches, camera cutouts, rounded corners, or system gesture bars that can overlap your app's content if you ignore them. SafeAreaView (or the more flexible useSafeAreaInsets hook from react-native-safe-area-context) automatically adds the right amount of padding so your content stays clear of these areas on every device, without you having to hardcode device-specific measurements. Think of it as automatically pulling your furniture back from a room's irregular alcoves and pillars, so nothing gets visually "swallowed" by the architecture.

JSX · SafeAreaView 
import { SafeAreaView } from 'react-native-safe-area-context';

<SafeAreaView style={{ flex: 1 }}>
  <Text>Never hidden behind a notch or status bar</Text>
</SafeAreaView>

What is KeyboardAvoidingView and what problem does it solve?

📖 Answer

When the on-screen keyboard pops up to let a user type into a TextInput, it can cover the very input field they're trying to use, especially near the bottom of the screen. KeyboardAvoidingView wraps your content and automatically shifts it up (via padding, position, or height adjustments, depending on the chosen behavior prop) so the active input stays visible above the keyboard. Without it, a form at the bottom of a screen becomes nearly unusable once the keyboard appears — like trying to write on a desk while a drawer slides shut right where your hand is.

JSX · KeyboardAvoidingView 
<KeyboardAvoidingView
  behavior={Platform.OS === 'ios' ? 'padding' : 'height'}
  style={{ flex: 1 }}>
  <TextInput placeholder="Type here…" />
</KeyboardAvoidingView>

What is the AppState API and what is it used for?

📖 Answer

The AppState API tells you whether your app is currently 'active' (in the foreground, visible), 'background' (minimized), or 'inactive' (transitioning, mostly an iOS-specific brief state), and lets you subscribe to changes between these states. It's commonly used to pause a video or refresh data the moment a user returns to the app, or to log a session-end timestamp the moment they leave. Think of it as a doorbell camera that tells you exactly when someone walks in or out of the room, so you can react at precisely the right moment.

JS · Listening for App State 
import { AppState } from 'react-native';

useEffect(() => {
  const sub = AppState.addEventListener('change', (nextState) => {
    if (nextState === 'active') refreshData(); // user just came back
  });
  return () => sub.remove();
}, []);

What is Deep Linking and how does it work in React Native?

📖 Answer

Deep linking lets a special URL (like myapp://product/42 or a real web link via Universal/App Links) open your app directly to a specific screen, rather than just launching to the home screen. Tapping a deep link from an email, text message, or another app passes that URL to your running or newly-launched app, where React Navigation (or Expo Router automatically) parses it and routes to the matching screen, often with parameters extracted from the URL itself. It's like a building having a direct elevator button for "Floor 12, Suite B" instead of every visitor having to start at the lobby and ask for directions.

JS · Linking Config 
const linking = {
  prefixes: ['myapp://', 'https://myapp.com'],
  config: {
    screens: { Product: 'product/:id' }, // myapp://product/42 → Product screen
  },
};

<NavigationContainer linking={linking}>...</NavigationContainer>

What is a WebView and when would you use one?

📖 Answer

react-native-webview embeds a full browser engine inside your app, capable of rendering any web page or HTML content right inside a native screen. It's used when you need to show content that's easier (or only possible) to render as a web page — a third-party payment checkout page, full rich-text/HTML content, an embedded video player from a service without a native SDK, or quickly wrapping an existing website inside an app shell. Think of it as a window built into your app's wall that looks directly out into the web, rather than rebuilding everything you see through that window using native components.

JSX · Basic WebView 
import { WebView } from 'react-native-webview';

<WebView source={{ uri: 'https://example.com' }} style={{ flex: 1 }} />

What is the Modal component, and how is it different from navigating to a regular screen?

📖 Answer

Modal renders content above the entire current screen, usually with a dimmed background behind it, without going through the normal navigation stack — it's meant for short-lived, focused interruptions like a confirmation dialog, an image preview, or a quick settings panel, rather than a full new "page" in your app's flow. Unlike pushing a new screen via React Navigation, a Modal doesn't add a back-button history entry by default and is typically dismissed by tapping outside it or an explicit close action rather than the system back gesture. It's the difference between a pop-up sticky note placed temporarily on your desk (Modal) versus turning to a whole new page in a notebook (screen navigation).

JSX · Basic Modal 
<Modal visible={showModal} transparent animationType="slide">
  <View style={{ flex: 1, justifyContent: 'center', backgroundColor: '#000a' }}>
    <View style={{ backgroundColor: '#fff', margin: 30, padding: 20 }}>
      <Text>Are you sure?</Text>
      <Button title="Close" onPress={() => setShowModal(false)} />
    </View>
  </View>
</Modal>

What is Jest and how is it used for testing React Native apps?

📖 Answer

Jest is a JavaScript test runner — it discovers test files, runs the functions inside them, and reports pass/fail results, and it ships pre-configured by default in every new React Native project. You write a test() or it() block describing what's being checked, call your code, and assert the output matches what you expect using expect(). It's used for "unit testing" pure logic — a price-formatting function, a validation rule, a reducer — where you can verify correctness without rendering any actual UI.

JS · Basic Jest Test 
function formatPrice(cents) {
  return `$${(cents / 100).toFixed(2)}`;
}

test('formats cents into a dollar string', () => {
  expect(formatPrice(1999)).toBe('$19.99');
});

What is React Native Testing Library, and what kind of testing does it enable?

📖 Answer

React Native Testing Library renders your actual components in a simulated environment (no real device needed) and lets you query the result the way a user would — finding text on screen, simulating a button press, then asserting on what's now visible — rather than poking at internal component state or implementation details directly. This is "component testing": verifying your UI behaves correctly from the outside, which tends to survive refactors better than tests tightly coupled to internal implementation.

JS · Component Test Example 
import { render, fireEvent, screen } from '@testing-library/react-native';

test('increments counter on press', () => {
  render(<Counter />);
  fireEvent.press(screen.getByText('Increment'));
  expect(screen.getByText('Count: 1')).toBeTruthy();
});

What is Detox, and what kind of testing does it provide?

📖 Answer

Detox is an end-to-end (E2E) testing framework that runs your actual compiled app on a real simulator/emulator or device and automates full user flows — tapping through a real login screen, navigating between real screens, verifying real navigation transitions happen correctly. Unlike Jest (logic only) or Testing Library (isolated components), Detox tests the entire app working together exactly as a real user would experience it, catching integration bugs that smaller-scoped tests would miss — at the cost of being slower and more complex to set up and run.

JS · Detox E2E Flow 
describe('Login flow', () => {
  it('logs in and shows the home screen', async () => {
    await element(by.id('email')).typeText('user@test.com');
    await element(by.id('password')).typeText('secret123');
    await element(by.id('loginButton')).tap();
    await expect(element(by.id('homeScreen'))).toBeVisible();
  });
});

What are environment variables in React Native, and why do you need them?

📖 Answer

Environment variables let you keep environment-specific values — API base URLs, public API keys, feature flags — out of your hardcoded source code, so you can swap them between development, staging, and production builds without changing a single line of app logic. In Expo apps, this is commonly handled through app.config.js's extra field plus EAS's secret environment variable storage for sensitive values used during the build process, ensuring secrets never get committed into your git repository.

JS · Reading an Env Variable 
// app.config.js
export default {
  expo: { extra: { apiUrl: process.env.API_URL } },
};

// In your app code:
import Constants from 'expo-constants';
const apiUrl = Constants.expoConfig.extra.apiUrl;

What is an EAS Update (OTA update), and how is it different from a full app store release?

📖 Answer

An EAS Update (Over-The-Air update) pushes a new version of your app's JavaScript bundle directly to users' devices, who download it automatically the next time they open the app — without going through App Store or Play Store review at all. This works for changes confined to your JS code (bug fixes, UI tweaks, content updates) but cannot ship changes that need new native code or permissions, which still require a full store submission and review cycle. It's the difference between quietly swapping out the contents of a vending machine overnight (OTA update) versus needing to physically replace the machine itself, which requires inspection and approval first (store release).

Bash · Publishing an OTA Update 
eas update --branch production --message "Fix checkout bug"
// Users get this update automatically — no App/Play Store review needed
// (only works for JS-only changes — native code changes need a new build)

What basic accessibility practices should every React Native developer follow?

📖 Answer

Give every meaningful interactive element an accessibilityLabel describing what it does (especially icon-only buttons with no visible text), use accessibilityRole so screen readers announce an element correctly ("button," "link," "header"), ensure tappable elements meet a minimum touch target size (roughly 44×44 points) so they're usable for people with motor impairments, and verify text/background color combinations have sufficient contrast for low-vision users. These small additions are what let screen readers like VoiceOver (iOS) and TalkBack (Android) describe your app coherently to users who can't see the screen — skipping them effectively locks those users out of your app entirely.

JSX · Basic Accessibility Props 
<Pressable
  accessibilityRole="button"
  accessibilityLabel="Add item to cart"
  style={{ minWidth: 44, minHeight: 44 }}
  onPress={addToCart}>
  <Icon name="cart-plus" />
</Pressable>
